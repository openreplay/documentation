Установка трекера OpenReplay в проект на основе Svelte относительно проста.

Ключом к тому, чтобы это заработало, будет использование контекста, который, хотя и звучит похоже на то, как мы делаем это с Next.js, реализация будет другой.

## SvelteKit и поведение SSR

Поскольку нашему трекеру для работы требуется контекст браузера, вам может понадобиться скорректировать пример «Создание контекста трекера», как показано ниже, чтобы убедиться, что он импортируется и создаётся на стороне клиента:

```jsx
<script>
  import { onMount } from 'svelte';

  onMount(async () => {
    if (process.browser) {
      const Tracker = await import('@openreplay/tracker');
      const tracker = new Tracker({
	      projectKey: "", // pKey
	      ingestPoint: "ingestPoint",
      })

      tracker.start()
    }
  })
</script>
```

## Использование контекста для обмена данными

В этом решении будет использоваться Context из Svelte, который позволяет обмениваться данными между родительским компонентом и всеми его потомками (во многом подобно провайдерам Context в React).

В этом конкретном примере мы будем добавлять трекер для раздела Admin нашего приложения, поэтому мы зададим контекст внутри файла layout-а Admin (`src/layout/Admin.svelte`). Затем мы запустим его на первой странице, отрендеренной внутри этого раздела.

## Создание контекста трекера

Внутри файла `src/layout/Admin.svelte`, внутри тега `script`, давайте добавим следующий код:

```jsx
import {Tracker, key} from "../context/tracker"

let tracker = new Tracker({
    projectKey: __myapp.env.OPENREPLAY_PROJECT_KEY
});

setContext(key, {
 getTracker: () => tracker
} )
```

Конечно, внутри этого файла у вас будет больше кода, но то, что вы видите выше, актуально для нас прямо сейчас.

1. Сначала мы импортируем Tracker из OpenReplay и переменную key (об этом чуть позже).
2. Затем мы создаём экземпляр Tracker. Как вы можете видеть, единственная опция конфигурации, которую мы используем, — это ключ проекта, который вы получите из интерфейса после настройки проекта в OpenReplay.
3. Мы сохраняем экземпляр трекера в контексте, созданном путём вызова `setContext`. Переменная `key` позже будет использоваться для извлечения этого конкретного значения из контекста.

💡**Примечание для пользователей self-hosted:** если вы используете self-hosted версию OpenReplay, вам также придётся указать свойство конфигурации `ingestPoint` на шаге 2. Это свойство задаёт конечную точку приёма (ingestion endpoint) данных трекера. Пользователям облака это не нужно, потому что по умолчанию трекер будет знать, где находится SaaS-версия этой конечной точки, но если
вы размещаете его самостоятельно, вам нужно будет указать его (это должно быть что-то вроде `https://openreplay.mydomain.com/ingest`).

## Использование контекста трекера

Использование контекста (и вызов метода `start`) так же просто, как получение контекста через функцию `getContext` из Svelte.

Контекст содержал объект с функцией под названием getTracker, поэтому мы выполним деструктуризацию и вызовем эту функцию.

```jsx
import {key} from "../../context/tracker"

let {getTracker} = getContext(key)
onMount( () => {
	let tracker = getTracker()
	tracker.start()
})
```

После этого остаётся лишь вызвать метод `start` изнутри коллбэка `onMount`. Мы делаем это, чтобы убедиться, что вызываем этот метод из браузера, иначе трекер не будет работать.

## Файл context/tracker

Мы использовали этот файл в обоих местах, поэтому пришло время быстро его рассмотреть.

Этот файл экспортирует два основных элемента:

1. Класс `Tracker`, экземпляр которого мы создали в файле layout-а.
2. Мы используем переменную `key` для сохранения и извлечения контекста.

Вот содержимое файла:

```jsx
import Tracker from '@openreplay/tracker'

const key = Symbol("openreplay tracker")

export {Tracker, key}
```

Единственный интересный момент здесь в том, что переменная `key` не является простой строкой, на самом деле это `Symbol`. Причина этого — избежать возможных коллизий имён с другими сторонними библиотеками, которые также используют контекст.

## Работа с переменными окружения (ENV)

Последнее, что нужно рассмотреть здесь, — это то, как мы обрабатываем ключ проекта, поскольку он не должен быть жёстко прописан в вашем коде.

Обычно для этого вы использовали бы переменную окружения, однако, поскольку нам нужны эти данные в браузере, а Svelte не предоставляет простого способа этого добиться, мы решим эту проблему с помощью двух пакетов: [dotevn](https://www.npmjs.com/package/dotenv) и [@rollup/plugin-replace](https://www.npmjs.com/package/@rollup/plugin-replace).

С одной стороны, мы будем использовать пакет dotenv, чтобы получить возможность определять наши переменные окружения внутри файла `.env`. Мы определим ту, которая нам нужна:

```jsx
OPENREPLAY_PROJECT_KEY="<YOUR PROJECT KEY>"
```

С другой стороны, поскольку мы не можем использовать `dotenv` в нашем коде фронтенда (потому что он зависит от модуля `fs`, который, очевидно, недоступен в браузере), мы будем использовать плагин replace для roll-up. Этот плагин заменит код в наших файлах на то, что мы ему укажем. Поэтому мы настроим плагин на замену строки `“__myapp.env.OPENREPLAY_PROJECT_KEY”` на фактический ключ проекта, загруженный из файла `.env` (благодаря пакету dotenv).

Чтобы это сделать, откройте файл `rollup.config.js` в корне проекта и добавьте следующий код в ключ `plugins`.

```jsx
replace({
  "__myapp.env.OPENREPLAY_PROJECT_KEY": `"${process.env.OPENREPLAY_PROJECT_KEY}"`
})
```

Этот плагин будет выполняться на стороне бэкенда во время процесса сборки (bundling), что делает это возможным. Если вы хотите изучить полное содержимое этого конфигурационного файла, вы можете [посмотреть его здесь](https://github.com/deleteman/openreplay-svelte-example/blob/main/rollup.config.js).

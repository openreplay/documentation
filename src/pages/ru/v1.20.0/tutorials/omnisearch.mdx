## Используем возможности Omnisearch

Верхняя строка поиска, которую вы видите в интерфейсе OpenReplay, — это гораздо больше, чем просто строка поиска: она позволяет создавать очень сложные поисковые запросы с минимальными усилиями.

Для целей примеров, показанных здесь, мы будем рассматривать тестовое приложение под названием «I'm bored», которое предлагает занятия для вас и вашей компании друзей на основе различных параметров поиска.

Вот скриншот интерфейса, с которым взаимодействовали бы наши вымышленные пользователи:

![](https://i.imgur.com/6BtPzTs.png)

## Выполнение простого поиска
Самый базовый сценарий использования строки поиска — это поиск сессий конкретного пользователя. Либо потому, что это ваш тестовый пользователь, либо потому, что вы ищете сессию клиента, который жалуется на ваш продукт.
Если вы задаёте ID пользователя при инициализации трекера, вы сможете видеть различные ID в списке сессий.

Для этого, когда вы нажимаете на строку поиска, выберите опцию «User Id», как показано ниже:

![](https://i.imgur.com/jZU6hwG.png)

Когда вы это сделаете, вы увидите опцию указать id пользователя в составе фильтров.

![](https://i.imgur.com/6Ac5cDB.png)

Отметим, что полный список фильтров, к которым у вас есть доступ, следующий:

- **Click**: фильтр по тому, _где_ пользователь нажимает.
- **Input**: фильтр по вводу, использованному пользователем.
- **Page**: фильтр по URI страницы, использованной во время сессии воспроизведения.
- **RevId**: фильтр по ID ревизии вашего приложения (только при использовании с SDK).
- **Referrer**: URL, с которого пользователь попадает в ваше приложение.
- **Duration**: фильтр воспроизведений по их продолжительности.
- **User** Country: фильтр по стране пользователя, взаимодействующего с вашим приложением (вычисляется по его публичному IP-адресу).
- **Custom Events**: фильтр по пользовательскому событию, запущенному из SDK.
- **Network Request**: фильтр по свойствам запроса, отправленного интерфейсом (payload, URL и т. д.).
- **GraphQL**: фильтр по свойствам запроса GraphQL (имя, метод и payload).
- **StateAction**: фильтр по действию над хранилищем состояния (т. е. Redux, VueX и т. д.).
- **Error**: фильтр по сообщению об ошибке (т. е. сообщению, использованному при выбрасывании исключения).
- **Issue**: фильтр по типу проблемы (Click Rage, Dead Click, Excessive Scrolling, Bad Request, Missing Resources и Memory).
- **User OS**: фильтр по операционной системе ваших пользователей.
- **User Browser**: фильтр по названию браузера, используемого вашим пользователем (фильтрует только по фактическим данным, поэтому если вы не можете найти результаты при поиске «Chrome», это означает, что никто из ваших пользователей его не использует).
- **User Device**: фильтр по мобильной версии устройства, используемого пользователем.
- **Platform**: фильтр по платформе, используемой вашим пользователем (Mobile, Tablet или Desktop).
- **User Id**: фильтр по ID вашего пользователя (заданному методом `setUserID`).
- **DOM complete**: фильтр по времени, которое требуется для срабатывания события DOM Complete.
- **Largest contentful paint**: фильтр по времени, которое требуется для срабатывания события DOM Complete.
- **Time to First Byte**: фильтр по времени, которое требуется для срабатывания [события TTFB](https://fr.wikipedia.org/wiki/Time_to_first_byte).
- **AVG CPU Load**: фильтрует по средней загрузке CPU в течение сессии.
- **AVG Memory Usage**: фильтрует по среднему использованию памяти в течение сессии.
- **Failed Request**: логический фильтр, отфильтровывает сессии, в которых нет неудавшихся запросов.

 

Далее, по умолчанию фильтр будет пытаться найти точное совпадение со словом, которое вы туда введёте. Это связано с модификатором «is» рядом с полем ввода. Вы можете изменить его на другое поведение, например:

- is any
- is not
- starts with
- ends with
- contains
- not contains
- is undefined

Для нашего примера мы оставим его на «is», поскольку нам известен точный адрес электронной почты нашего пользователя.
Как вы можете видеть на следующем изображении, когда мы фильтруем по id «thisisatest@gmail.com», мы получаем только эти результаты:

![](https://i.imgur.com/kAjALWs.png)

Но что, если нам нужно больше?

## Добавление нескольких шагов в наш поиск
Что, если мы хотим искать сессии, в которых пользователь нажимает на определённую ссылку, затем приложение отправляет запрос к определённому endpoint, и, наконец, мы получаем от сервера определённый код ответа?

По отдельности каждое условие не так уж сложно, но мы ищем их все вместе.
Это не проблема для нашего Omnisearch.
В левом нижнем углу секции фильтров у вас есть опция «+ADD STEP» для добавления нескольких факторов к фильтрам.

Когда вы нажмёте на неё, вы увидите те же опции, что и при нажатии на строку поиска.
Здесь вы сможете выбрать все опции в соответствии с тем, как, по вашему мнению, следует построить поиск.

В нашем случае мы хотим найти:

- Конкретный id пользователя.
- Все сессии с нажатием на определённую ссылку.
    - Все сессии с запросами к одному определённому URL.
- Наконец, все сессии, имеющие одно и то же семейство кодов статуса.

Когда вы свяжете все эти фильтры вместе, вы получите сложный поисковый фильтр, который соответствует вашим конкретным сессиям.
Давайте посмотрим, как это выглядело бы в интерфейсе:

![](https://i.imgur.com/RX9PeH3.png)

Здесь стоит отметить несколько моментов:

1. ID пользователя обрабатывается как обычный фильтр, тогда как остальные параметры обрабатываются как события. Это связано с тем, что вы также можете указать порядок, в котором происходят эти события (а id пользователя всегда будет присутствовать).
2. Событие нажатия указывает текст на кнопке, на которую мы нажимаем (обратите внимание на строку «Get me something!»). Это будет **автоматически дополняться** по мере того, как вы начнёте печатать.
3. Модификатор для URL изменён с «is» на «contains». Это важно, потому что автодополнение не будет учитывать параметры URL, но в рамках этого запроса мы отправляем все атрибуты формы в виде параметров URL (т. е. это GET-запрос). Таким образом мы можем охватить все запросы независимо от используемой комбинации.
4. Мы изменили модификатор для значения «status code» с `=` на `<`, потому что я ищу все коды успешного ответа (2XX).
5. Обратите внимание на «AND», выделенный в правом верхнем углу рядом с «OR» и «THEN». Этот модификатор делает так, что все шаги и фильтры оцениваются с помощью AND. Мы могли бы изменить их на OR (например), если бы хотели, чтобы они все оценивались по отдельности.

Вы можете менять фильтры или даже логические условия, чтобы сделать ваш фильтр именно таким, как вам нужно.

Но что, если ваш поиск требует нескольких шагов? Что, если вы ищете сессии, которые соответствуют определённой воронке (funnel)? Давайте посмотрим, как можно настроить многошаговую воронку с помощью поисковых фильтров.

## Создание воронки из поисковых фильтров

Воронка — это не более чем набор шагов, которые ваши пользователи должны пройти, чтобы добраться до определённой точки. Обычно, когда это происходит, некоторые пользователи не доходят до конца, возможно, потому что они потеряли интерес, или, возможно, из-за ошибок в вашем приложении.
И именно во втором сценарии OpenReplay может помочь вам выявить проблемы.

Первое, что нужно сделать, — это определить воронку с помощью OmniSearch. Для этого просто укажите необходимые шаги, используя многочисленные фильтры, перечисленные выше.

Для нашего примера я буду искать сессии, в которых происходит следующее:

- Пользователь посещает домашнюю страницу моего приложения (URI равен «/»).
- Пользователь взаимодействует с полем ввода «first name».
- Затем пользователь нажимает на ссылку «second page».
- Пользователь переходит на другую страницу с URI «/second-page».
- Затем пользователь нажимает на ссылку «Home page».

Как вы можете видеть, список шагов может быть любой произвольной комбинацией, имеющей смысл в рамках вашего приложения.
Этот конкретный пример выглядел бы так:

![](https://i.imgur.com/uEx2yGZ.png)

Сам по себе этот поиск перечислил бы сессии, соответствующие шаблону. Однако мы можем пойти немного дальше, сохранив его как воронку, а затем проанализировав информацию, собранную OpenReplay.

Нажатие на «Save funnel» предложит вам добавить имя для вашей воронки, и она появится в нижнем левом углу экрана.

![](https://i.imgur.com/bt5O08M.png)

И как только вы войдёте в воронку, вы получите обзор того, что происходит во время воронки:

![](https://i.imgur.com/cgyJaKO.png)

Как вы можете заметить, в этом примере мы видим проблемы с Dead Clicks и Click Rages в 3 разных сессиях.

## Сохранение ваших фильтров
Эта функциональность гарантирует, что вам не придётся заново создавать ваши сложные фильтры каждый раз, когда вы захотите их использовать.
И сделать это чрезвычайно просто.

Допустим, у нас есть фильтр, который показывает только сессии с dead clicks (нажатиями, которые ничего не делают) на домашней странице. Мы хотим сохранить этот поиск, чтобы постоянно проверять, появляются ли в будущем новые подобные проблемы.
Всё, что нам нужно сделать, — это настроить его и нажать на ссылку «SAVE SEARCH» в нижнем правом углу поля поиска.
Как показано на следующем изображении:

![](https://i.imgur.com/b3GP8xh.png)

Дайте ему имя и нажмите на кнопку «Create».

![](https://i.imgur.com/TONFroV.png)

После этого ваш сохранённый поиск появится рядом со строкой поиска, вот так:

![](https://i.imgur.com/n1GsndD.png)

Omnisearch предоставляет вам поисковые суперспособности, но чтобы воспользоваться ими, вы должны выбирать правильные опции.

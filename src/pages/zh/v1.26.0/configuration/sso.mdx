单点登录（通过 SAML2）仅在 OpenReplay Enterprise Edition 中可用。

## 身份提供商 (IDP) 配置

在你的身份提供商的控制台中，创建一个名为 `openreplay` 的新应用（你可以使用此<a href="https://docs.openreplay.com/static/OpenReplay.png" target="_blank">图标</a>）。在配置页面中，请确保设置以下值（请务必将 <code>{initialValues.domain}</code> 替换为正确的值，例如 `https://openreplay.mycompany.com`）：

| 变量 | 值 |
|----------|-------------|
| ACS URL（也称为 Single Sign On URL 或 Consumer URL） | <code><ORDynamicCode id="domain" initialValue={initialValues.domain} client:idle/>/api/sso/saml2/acs/</code> |
| Entity ID（也称为 Audience） | <code><ORDynamicCode id="domain" initialValue={initialValues.domain} client:idle/>/api/sso/saml2/metadata/</code> |
| Single Logout URL（也称为 SLO URL） | <code><ORDynamicCode id="domain" initialValue={initialValues.domain} client:idle/>/api/sso/saml2/sls/</code>（可选） |
| Name ID（有时可在 'Attribute Statements' 或 'Parameters' 部分中配置） | `Email` 或 `EmailAddress` 或 `urn:oasis:names:tc:SAML:1.1:nameid-format:emailAddress`，取决于你的身份提供商 |
| SAML initiator | 将其设置为 `Service Provider`（可选） |

在 'Attribute Statements' 或 'Parameters' 部分中，请确保定义以下字段：

| 字段 | 值 |
|----------|-------------|
| tenantKey | <code><ORDynamicCode id="tenantKey" initialValue={initialValues.tenantKey} client:idle/></code> 其值可在 Dashboard 中的 'Preferences' > 'Account' 下找到（如果你使用的是我们的 Cloud Dedicated 计划，则不得添加此属性） |
| firstName | `user.firstName` |
| lastName |  `user.lastName` |
| internalId | 内部 ID 或 `user.email`（用户在你的身份管理系统中的 ID）（可选） |
| groups | 表示用户在 OpenReplay 中的角色和管理员权限，取决于你的身份提供商，它可以是 `user.role` 或 `group` 等（可选，默认为 `member`，无管理员权限）。为了简化从 IdP 端管理 OpenReplay 角色，你可以创建一个名为 OpenReplay-admin 的组，将具有管理员权限的用户放入其中，并创建一个名为 OpenReplay-member 的组，将具有 'member' 角色的用户放入其中 |

现在你应该已经拥有下一步/部分所需的所有值。

## SSO (SAML2) 配置

如果你使用的是我们的 Cloud dedicated 计划，请前往[控制台](https://console.openreplay.com)，然后点击实例并在 SSO 部分下设置变量。

否则，如果你运行的是我们的 Enterprise Edition，请运行 `openreplay -e`，然后取消注释并更新 `chalice` 部分中以下的环境变量：

| 变量 | 描述 |
|----------|-------------|
| idp_entityId | 你的身份提供商的 `entityId`，也称为 `Issuer URL` |
| idp_sso_url | 你的身份提供商的 `singleSignOnService`，也称为 `SAML 2.0 Endpoint (HTTP)` |
| idp_x509cert | `x509cert`，必须是单行字符串，不含换行符。你可以使用<a href="https://www.samltool.com/format_x509cert.php" title="FORMAT A X509 CERTIFICATE" target="_blank">此工具</a>来格式化你的值 |
| idp_name | 身份提供商的名称（可选） |
| idp_sls_url | 你的身份提供商的 `singleLogoutService`，也称为 `SLO Endpoint (HTTP)`（可选） |
| enforce_SSO | 如果设置为 `true`，所有使用用户名和密码的连接都将被阻止（可选，默认为 `false`） |
| idp_group_prefixes | IdP 组名的前缀（可选，默认为 `OpenReplay-,or-`） |

然后，使用 `:wq` 保存并退出，以使更改生效：

## OpenReplay 证书

这是可选的，你可以添加一个 x509 证书以提供额外的 SSO 安全性：

1. 在你的服务器或安全机器上运行此命令（你可以根据需要更改证书的有效期）：
```bash
openssl req -x509 -nodes -days 365 -newkey rsa:2048 -keyout openreplay-sso.key -out openreplay-sso.crt
```
2. 在你的服务器上，运行 `openreplay -e`
3. 在 `chalice` 部分下，添加以下环境变量：
- **sp_crt**：必须是生成的 'openreplay-sso.crt' 文件的单行字符串，不含换行符（你可以使用<a href="https://www.samltool.com/format_x509cert.php" title="FORMAT A X509 CERTIFICATE" target="_blank">此工具</a>来格式化你的值）
- **sp_key**：必须是生成的 'openreplay-sso.key' 文件的单行字符串，不含换行符
4. 使用 `:wq` 保存并退出，以使更改生效
5. 将生成的 'openreplay-sso.crt' 文件添加到你的身份提供商的应用配置中

## 使用 Okta 的示例

1. 登录到你的 Okta 管理控制台并前往 'Applications' > 'Applications'
2. 按下 'Create new app integration'，然后选择 SAML 2.0 并按下 'Next'
3. 将 'App Name' 设置为 OpenReplay（你可以为你的应用上传此<a href="https://docs.openreplay.com/static/OpenReplay.png" target="_blank">图标</a>），然后按下 'Next'
4. 设置：
 - **Single sign on URL** 为 <code><ORDynamicCode id="domain" initialValue={initialValues.domain} client:idle/>/api/sso/saml2/acs/</code>
 - **Audience URI (SP Entity ID)** 为 <code><ORDynamicCode id="domain" initialValue={initialValues.domain} client:idle/>/api/sso/saml2/metadata/</code>
 - **Name ID format** 为 `EmailAddress`
5. 按下 Next，选择 'This is an internal app that we have created'，然后按下 'Finish'
6. 在 Sign On 选项卡中，向下滚动并按下 'Show legacy configuration'，然后按下 'Edit'
7. 在 'Profile Attribute Statements' 中定义以下字段：
 - **tenantKey**：格式为 `Basic`，并将值设置为 <code><ORDynamicCode id="tenantKey" initialValue={initialValues.tenantKey} client:idle/></code>，可在 OpenReplay 控制台的 'Preferences' > 'Account' 下找到（如果你使用的是我们的 Cloud Dedicated 计划，则不得添加此属性）
 - **firstName**：格式为 `Basic`，并将值设置为 `user.firstName`
 - **lastName**：格式为 `Basic`，并将值设置为 `user.lastName`
 - **internalId**：格式为 `Basic`，并将值设置为 `user.email`
8. 在 'Group Attribute Statement' 中定义以下字段（这是可选的）：
 - **groups**：格式为 `Basic`，过滤器为 `Match Regex`，值为 `OpenReplay-*`（或者你可以根据需要指定不同的过滤器和正则表达式）
9. 按下 'save'，向上滚动到右侧并按下 'View Setup Instructions' 以查看你的 SAML2 配置

如果你使用的是我们的 Cloud dedicated 计划，请前往[控制台](https://console.openreplay.com)，然后点击实例并在 SSO 部分下设置变量。

否则，如果你运行的是我们的 Enterprise Edition：

9.  通过 SSH 连接到你的 OpenReplay 安装并运行 `openreplay -e`
10.  在 `chalice` 部分下，取消注释，然后设置以下环境变量：
   - **idp_entityId**：Identity Provider Issuer
   - **idp_sso_url**：Identity Provider Single Sign-On URL
   - **idp_x509cert**：X.509 Certificate，必须是单行字符串，不含换行符（你可以使用<a href="https://www.samltool.com/format_x509cert.php" title="FORMAT A X509 CERTIFICATE" target="_blank">此工具</a>来格式化你的值）
   - **idp_name**：Okta
11.  最后，使用 `:wq` 保存并退出，以使更改生效

## 使用 Azure Active Directory 的示例

1. 登录到你的 Azure 门户并前往 'Azure Active Directory'
2. 按下 '+ Add'，选择 'Enterprise application'，然后点击 'Create your own application'
3. 将 'App Name' 设置为 OpenReplay，然后选择 'Integrate any other application you don't find in the gallery (Non-gallery)' 并按下 'Create'
4. 导航到左侧菜单中的 'Single sign-on'，并在 'Basic SAML Configuration' 块中设置以下值：
 - **Identifier (Entity ID)** 为 <code><ORDynamicCode id="domain" initialValue={initialValues.domain} client:idle/>/api/sso/saml2/metadata/</code>
 - **Reply URL (Assertion Consumer Service URL)** 为 <code><ORDynamicCode id="domain" initialValue={initialValues.domain} client:idle/>/api/sso/saml2/acs/</code>
 - **Logout URL** 为 <code><ORDynamicCode id="domain" initialValue={initialValues.domain} client:idle/>/api/sso/saml2/sls/</code>
5. 在 'Attribute & Claims' 块中添加以下 claims，并确保每个 claim 的 'Namespace' 字段都留空：
 - **tenantKey**：将值设置为 <code><ORDynamicCode id="tenantKey" initialValue={initialValues.tenantKey} client:idle/></code>，可在 OpenReplay 控制台的 'Preferences' > 'Account' 下找到（如果你使用的是我们的 Cloud Dedicated 计划，则不得添加此属性/claim）
 - **firstName**：将值设置为 `user.givenname`
 - **lastName**：将值设置为 `user.surname`
 - **internalId**：将值设置为 `user.mail`
6. 导航到 'Users and groups' 并将你选择的用户分配给该应用

如果你使用的是我们的 Cloud dedicated 计划，请前往[控制台](https://console.openreplay.com)，然后点击实例并在 SSO 部分下设置变量（值可在 'Azure Active Directory' > 'OpenReplay' > 'Single sign-on' 中找到）。

否则，如果你运行的是我们的 Enterprise Edition：

7. 通过 SSH 连接到你的 OpenReplay 安装并运行 `openreplay -e`
8. 在 `chalice` 部分下，取消注释，然后设置以下环境变量（值可在 'Azure Active Directory' > 'OpenReplay' > 'Single sign-on' 中找到）：
- **idp_entityId**：Azure AD Identifier
- **idp_sso_url**：Login URL
- **idp_sls_url**：Logout URL
- **idp_x509cert**：X.509 Certificate，必须是单行字符串，不含换行符（你可以使用<a href="https://www.samltool.com/format_x509cert.php" title="FORMAT A X509 CERTIFICATE" target="_blank">此工具</a>来格式化你的值）
- **idp_name**：Active Directory
9.  最后，使用 `:wq` 保存并退出，以使更改生效

## SCIM（跨域身份管理系统）配置

Openreplay 支持在 Enterprise Edition 中预配用户身份的选项。

要与 Openreplay 的 SCIM 服务集成，只需在你的身份提供商控制台中启用 SCIM 预配。请务必设置以下值以实现顺畅的集成：

| 变量 | 描述 |
|----------|-------------|
| SCIM connector base URL | <code><ORDynamicCode id="domain" initialValue={initialValues.domain} client:idle/></code>/api/sso/scim/v2/ |
| User unique identifier field name | `email` |
| Support provisioning actions | Push new users, Push profile updates, Push groups |
| Authentication mode | OAuth2 |
| Access token endpoint | <code><ORDynamicCode id="domain" initialValue={initialValues.domain} client:idle/></code>/api/sso/scim/v2/token/ |
| Authorization endpoint | <code><ORDynamicCode id="domain" initialValue={initialValues.domain} client:idle/></code>/api/sso/scim/v2/authorize/ |
| Client id | <code><ORDynamicCode id="tenantKey" initialValue={initialValues.tenantKey} client:idle/></code> 其值可在 Dashboard 中的 'Preferences' > 'Account' 下找到 |
| Client secret | 其值可通过在你的 Postgres 数据库中执行此查询找到：<code>SELECT tenant_secret FROM tenants WHERE tenant_key = '<code><ORDynamicCode id="tenantKey" initialValue={initialValues.tenantKey} client:idle/></code>';</code> |

设置这些值后，请按照你的身份提供商控制台中的提示来测试集成并授权该应用。

完成此操作后，你将能够使用 Openreplay 作为 SCIM 服务提供商来预配用户身份。

> **注意：** 虽然 Openreplay 充当 SCIM 服务提供商，但它不支持将用户划分到多个组中（即每个用户只能属于一个组）。如果不遵守此规则，则用户可能会被分配到比预期更少或更多的项目，并且在 Openreplay 应用中可能拥有更少或更多的权限。

### 使用 Okta 的示例

在你的 SAML 2.0 应用中，执行以下操作

#### 添加 SCIM 预配

1. 点击 **General** 选项卡
2. 在 **App Settings** 部分，点击 **Edit**
3. 在 **Provisioning** 字段中，选择 **SCIM**，然后点击 **Save**

#### 选择预配选项

1. 点击 **Provisioning** 选项卡。SCIM 连接设置将出现在 **Settings > Integration** 下。
2. 在 **Settings > Integration** 中，点击 **Edit**。
3. 指定上面提到的 **SCIM connector base URL** 和用户唯一标识符的字段名称。
4. 在 **Supported provisioning actions** 下，选择上面提到的预配操作。
5. 将 **Authentication Mode** 下拉菜单设置为上面提到的值。
6. 按照上面所述设置 **Access token endpoint**、**Authorization endpoint**、**Client id** 和 **Client secret**。
7. 点击 **Save**。
8. 点击 **Authenticate application**。
9. 在 **Provisioning** 选项卡中，点击 **To App**。
10. 点击 **Edit**，启用 `Create Users`、`Update User Attributes`、`Deactivate Users` 并保存更改（确保在按下保存后这些选项处于选中状态，否则刷新页面并重新执行此步骤）。
- 在开始下一步之前，我们建议在 Okta 中创建名为 `OpenReplay-admin` 的新组（其中列出在 OpenReplay 中具有管理员权限的用户）和名为 `OpenReplay-members` 的新组（其中列出在 OpenReplay 中具有 `member` 角色的用户），...    
11. 前往 **Push Groups** 选项卡，点击 **+ Push Groups**（按名称或按规则），选择你想要推送到 OpenReplay 的组，然后按下 **save**
- 此时，推送的组代表 OpenReplay 中的角色和管理员权限，但不包含为 OpenReplay 预配的用户列表。
12. 前往 **Assignmets** 选项卡，并以适合你的方式添加用户（使用人员或组）

## 错误

### 响应在 http 而非 https 上收到

这种情况非常常见，是由于请求中缺少 'X-Forwarded-Proto' 标头所致。为了解决该问题，请确保 Load Balancer 正确地将该标头转发到 OpenReplay 后端，并运行 `openreplay -e`，在 `ingress-nginx > config` 部分下，取消注释 `use-forwarded-headers: true` 这一行。

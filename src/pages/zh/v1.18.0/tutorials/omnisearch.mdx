## 充分利用 Omnisearch

你在 OpenReplay 界面上看到的顶部搜索栏远不止是一个简单的搜索栏，它让你能够以极少的精力构建非常复杂的搜索查询。

为了说明这里展示的示例，我们将讨论一个名为 "I'm bored" 的测试应用程序，它会根据不同的搜索参数为你和你的朋友们推荐活动。

下面是我们的虚拟用户将要与之交互的界面截图：

![](https://i.imgur.com/6BtPzTs.png)

## 执行简单搜索
搜索栏最基本的使用场景，就是搜索某个特定用户的会话。可能因为那是你的测试用户，也可能因为你正在查找某个抱怨你产品的客户的会话。
如果你在实例化 tracker 时设置了用户 ID，你就能在会话列表中看到不同的 ID。

要做到这一点，当你点击搜索栏时，选择 "User Id" 选项，如下所示：

![](https://i.imgur.com/jZU6hwG.png)

当你这样做时，你会看到将用户 id 指定为过滤条件之一的选项。

![](https://i.imgur.com/6Ac5cDB.png)

需要说明的是，你能够使用的完整过滤器列表如下：

- **Click**：按用户点击的_位置_进行过滤。
- **Input**：按用户使用的输入进行过滤。
- **Page**：按回放会话期间所使用页面的 URI 进行过滤。
- **RevId**：按你应用的修订 ID 进行过滤（仅在与 SDK 一起使用时）。
- **Referrer**：用户到达你应用程序的来源 URL。
- **Duration**：按回放的时长进行过滤。
- **User** Country：按与你应用交互的用户所在国家进行过滤（根据其公共 IP 地址计算得出）。
- **Custom Events**：按从 SDK 触发的自定义事件进行过滤。
- **Network Request**：按界面发出的请求的属性进行过滤（payload、URL 等）。
- **GraphQL**：按 GraphQL 查询的属性进行过滤（名称、方法和 payload）。
- **StateAction**：按对状态存储的操作进行过滤（即 Redux、VueX 等）。
- **Error**：按错误消息进行过滤（即抛出异常时使用的消息）。
- **Issue**：按问题类型进行过滤（Click Rage、Dead Click、Excessive Scrolling、Bad Request、Missing Resources 和 Memory）。
- **User OS**：按你的用户的操作系统进行过滤。
- **User Browser**：按你的用户所使用浏览器的名称进行过滤（仅按实际数据过滤，因此如果你搜索 "Chrome" 找不到结果，则意味着没有用户在使用它）。
- **User Device**：按用户所使用设备的移动版本进行过滤。
- **Platform**：按你的用户所使用的平台进行过滤（Mobile、Tablet 或 Desktop）。
- **User Id**：按你的用户的 ID 进行过滤（使用 `setUserID` 方法设置）。
- **DOM complete**：按 DOM Complete 事件被触发所需的时间进行过滤。
- **Largest contentful paint**：按 DOM Complete 事件被触发所需的时间进行过滤。
- **Time to First Byte**：按 [TTFB 事件](https://fr.wikipedia.org/wiki/Time_to_first_byte) 被触发所需的时间进行过滤。
- **AVG CPU Load**：按会话期间的平均 CPU 负载进行过滤。
- **AVG Memory Usage**：按会话期间的平均内存使用量进行过滤。
- **Failed Request**：布尔值，过滤出没有失败请求的会话。

 

继续往下，默认情况下过滤器会尝试精确匹配你输入的词语。这是因为输入字段旁边有 "is" 修饰符。你可以将其更改为其他行为，例如：

- is any
- is not
- starts with
- ends with
- contains
- not contains
- is undefined

对于我们的示例，由于我们知道用户的确切电子邮件地址，我们将保留 "is"。
正如你在下图中看到的，当我们按 id "thisisatest@gmail.com" 进行过滤时，我们只得到那些结果：

![](https://i.imgur.com/kAjALWs.png)

但如果我们想要更多呢？

## 为我们的搜索添加多个步骤
如果我们想搜索这样的会话：用户点击某个特定链接，然后应用向某个特定端点发送请求，最后我们从服务器收到一个特定的响应代码，该怎么办？

单独来看，每个条件都不那么复杂，但我们要把它们全部放在一起来查找。
这对我们的 Omnisearch 来说不是问题。
在过滤器区域的左下角，你有 "+ADD STEP" 选项，可以向过滤器添加多个因素。

当你点击它时，你会看到与点击搜索栏时相同的选项。
在这里你可以按照你认为应该构建搜索的方式来选择所有选项。

在我们的例子中，我们想搜索：

- 某个特定的用户 id。
- 所有点击了某个特定链接的会话。
    - 所有向某个特定 URL 发出请求的会话。
- 最后，所有共享同一类状态代码的会话。

当你把所有这些过滤器组合在一起时，你就得到了一个复杂的搜索过滤器，它能匹配你特定的那些会话。
让我们看看这在界面中会是什么样子：

![](https://i.imgur.com/RX9PeH3.png)

这里有几点需要注意：

1. 用户 id 被当作普通过滤器处理，而其余参数被当作事件处理。这是因为你还可以指定这些事件发生的顺序（而用户 id 将始终存在）。
2. 点击事件指定了我们所点击按钮上的文本（注意 "Get me something!" 这个字符串）。一旦你开始输入，它就会**自动补全**。
3. URL 的修饰符从 "is" 改为了 "contains"。这一点很重要，因为自动补全不会考虑 URL 参数，但作为此请求的一部分，我们将表单的所有属性作为 URL 参数发送（即这是一个 GET 请求）。这样一来，无论使用何种组合，我们都能涵盖所有请求。
4. 我们将 "status code" 值的修饰符从 `=` 改为了 `<`，因为我要查找所有成功响应代码（2XX）。
5. 注意右上角 "OR" 和 "THEN" 旁边高亮显示的 "AND"。该修饰符使得所有步骤和过滤器都以 AND 进行求值。我们可以将它们改为 OR（例如），如果我们希望它们全部被单独求值的话。

你可以更改过滤器，甚至更改布尔条件，使你的过滤器完全符合你的需要。

但如果你的搜索需要多个步骤呢？如果你正在查找符合某个漏斗（funnel）的会话呢？让我们看看如何用搜索过滤器设置一个多步骤漏斗。

## 从搜索过滤器创建漏斗

漏斗无非就是你的用户为了到达某个点而必须经历的一系列步骤。通常当这种情况发生时，一些用户不会走完全程，也许是因为他们失去了兴趣，也许是因为你应用中的错误。
而正是针对第二种情况，OpenReplay 可以帮助你识别问题。

首先要做的是使用 OmniSearch 定义漏斗。要做到这一点，只需使用上面列出的众多过滤器说明所需的步骤即可。

对于我们的示例，我将搜索发生以下情况的会话：

- 用户访问我应用的主页（URI 为 "/"）。
- 用户与 "first name" 输入字段交互。
- 用户随后点击 "second page" 链接。
- 用户被带到 URI 为 "/second-page" 的另一个页面。
- 用户随后点击 "Home page" 链接。

正如你所见，步骤列表可以是在你应用中有意义的任意随机组合。
这个特定示例看起来会是这样：

![](https://i.imgur.com/uEx2yGZ.png)

单凭其本身，这次搜索会列出符合该模式的会话。然而，我们可以更进一步，将其保存为漏斗，然后分析 OpenReplay 收集到的洞察。

点击 "Save funnel" 会提示你为漏斗添加一个名称，它会出现在屏幕左下角。

![](https://i.imgur.com/bt5O08M.png)

而一旦你进入漏斗，你将获得漏斗期间所发生情况的概览：

![](https://i.imgur.com/cgyJaKO.png)

正如你所看到的，在这个示例中我们可以看到 3 个不同会话中出现了 Dead Clicks 和 Click Rages 的问题。

## 保存你的过滤器
此功能确保你不必每次想使用复杂过滤器时都重新创建它们。
而且这样做极其简单。

假设我们有一个过滤器，它只显示主页上有 dead clicks（不产生任何效果的点击）的会话。我们想保留这个搜索，以便能够持续检查未来是否出现类似的新问题。
我们需要做的，就是设置好它，然后点击搜索框右下角的 "SAVE SEARCH" 链接。
如下图所示：

![](https://i.imgur.com/b3GP8xh.png)

给它起一个名字，然后点击 "Create" 按钮。

![](https://i.imgur.com/TONFroV.png)

之后，你保存的搜索将出现在搜索栏旁边，就像这样：

![](https://i.imgur.com/n1GsndD.png)

Omnisearch 为你提供了搜索超能力，但要充分利用它们，你必须选对选项。

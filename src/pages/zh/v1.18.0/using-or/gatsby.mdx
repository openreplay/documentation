将 OpenReplay 追踪器安装到基于 Gatsby 的项目中相对简单。

![Sesion replay of a Gatsby site](/images/gatsby-ss.png)

考虑到 Gatsby 底层使用的是 React，我们要做的就是把追踪器代码添加到主页面中，并使用 `useEffect` 钩子调用 `start` 方法。

让我们来看看。

## 为什么要在静态网站上设置追踪器？

应用中缺少动态内容并不意味着你就不想利用 OpenReplay 的其他一些功能。

虽然你不会用会话回放来查找 bug，但你可以捕获有价值的指标，并围绕这些指标创建[你自己的自定义仪表盘](https://docs.openreplay.com/tutorials/custom-dashboard)。

你可以使用性能指标、web vitals 甚至与资源相关的指标，从而对用户在你网站上的整体体验有一个大致的了解。

## 在 Gatsby 中设置追踪器

假设有一个静态的 Gatsby 网站，其入口点是 `pages` 文件夹中的 `index.tsx` 文件，你需要向其中添加以下代码：

```jsx
import Tracker from '@openreplay/tracker';
const tracker = new Tracker({
  projectKey: process.env.GATSBY_OPENREPLAY_PROJECT_KEY
});

const Index = ({data, location }) => {

  React.useEffect(() => {
    tracker.start();
  }, [])

   //the rest of your code here...
}
```

追踪器在页面编译时被实例化（无论是 SSR 还是客户端渲染），但 start 方法只能从浏览器中调用，因此我们需要确保它在组件挂载之后被调用（因此这里使用了 `useEffect` 钩子）。`useEffect` 钩子的第二个参数为空数组，可确保回调只执行一次。

💡**给自托管（self-hosted）用户的提示：** 如果你使用的是 OpenReplay 的自托管版本，在实例化追踪器时还需要指定 `ingestPoint` 配置属性。该属性指定了追踪器数据的接收端点（ingestion endpoint）。云版用户不需要这样做，因为默认情况下追踪器知道该端点的 SaaS 版本在哪里，但如果你是自托管，则需要自行指定（它应该类似于 `https://openreplay.mydomain.com/ingest`）

## 在 Gatsby 中处理 ENV 变量

作为一项安全措施，我们建议不要将项目密钥硬编码到代码中。这意味着你需要将其导出为 ENV 变量，以便 Gatsby 读取它。

这样，OpenReplay 的项目密钥就会存放在一个名为 `GATSBY_OPENREPLAY_PROJECT_KEY` 的变量中。

请注意，该名称以 `GATSBY_` 为前缀，这会告诉 Gatsby 确保该变量在前端代码中也可用。否则，除非你正在执行 Node.js 代码，否则你将无法读取该值。

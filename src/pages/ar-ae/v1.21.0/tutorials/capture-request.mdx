في بعض الأحيان لا تكون الأخطاء في الكود الخاص بالعميل (client) واضحة بقدر ما تكون عندما تحصل على شاشة فارغة لأن أيًّا من كود JavaScript الخاص بك لا يعمل.
وفي بعض الأحيان تكون المشكلة في تطبيقك أنه عند بعض الإجراءات التي يمكن لمستخدمك القيام بها، فإنك تكوّن طلبًا إلى الخادم بشكل غير صحيح.
كود العميل يعمل، ولا تحصل على أي أخطاء JS في وحدة التحكم (console)، لكن الواجهة الخلفية (back-end) لا تستسيغ حقًّا ما ترسله إليها.

باستخدام OpenReplay، يمكنك التقاط الاتصال بين العميل والخادم كجزء من إعادة تشغيل الجلسة القياسية لديك، ومراجعته لاحقًا. فلنلقِ نظرة إذًا على كيفية القيام بذلك ونوع الفائدة التي يمكننا الحصول عليها منه.

## تطبيق نموذجي
لأغراض هذا الدليل الإرشادي، أنشأتُ تطبيق React بسيطًا يستخدم [Bored API](https://www.boredapi.com/). إنها واجهة برمجة تطبيقات بسيطة جدًّا تُرجع اقتراح نشاط عشوائي استنادًا إلى بعض المعطيات.
لذا أنشأتُ "I'm bored App"، الذي يبدو هكذا:

![](https://i.imgur.com/ZCEOtiC.png)

ويمكنك العثور عليه مباشرًا على Netlify [هنا](https://venerable-tartufo-f87748.netlify.app)، أو إذا أردت الاطلاع على الكود لمعاينته بالتفصيل، فهو متاح بالكامل على [GitHub](https://github.com/deleteman/im-bored-app).

يتكوّن هذا التطبيق من مكوّنين (components)، مكوّن `SearchForm` الذي يتولى عرض هذين الحقلين والزر، بالإضافة إلى إرسال الطلب الفعلي إلى الـ API.
ومكوّن `Suggestion` الذي يكتفي بعرض الاقتراح داخل صندوق أنيق المظهر.

سأركّز على المكوّن الأول، لأنه الوحيد الذي يرسل الطلبات باستخدام دالة `fetch`.

**لاحظ** أن هذه التقنية تعمل أيضًا مع أي طلب يُنفَّذ باستخدام **Axios** كذلك.

فلنلقِ نظرة سريعة على المكوّن لفهم ما يفعله.

### كود مكوّن SearchForm

هذا ليس مكوّنًا معقّدًا، لكن هناك قسمًا وثيق الصلة على نحو خاص بحالة الاستخدام هذه تحديدًا، فلنلقِ عليه نظرة سريعة.

```javascript
import { Container, Col, Form, Row, Button } from 'react-bootstrap';

const SearchForm = ({setResult, fetcher}) => {
const getSomething = async (evt) => {
    evt.preventDefault()
    let form = evt.target

    const API_URL = "/api/activity?"
    let getParams = {}
    if(form.participants.value !== '') {
      getParams.participants = form.participants.value
    }

    if(form.priceRange.value !== '') {
      let prices = form.priceRange.value.split("_")
      getParams.minprice = prices[0]
      getParams.maxprice = prices[1]
    }

    let results = await fetcher(API_URL + new URLSearchParams(getParams), {
        mode: 'no-cors'
    })
    setResult(await results.json())

    return false
  }

    return (
        <Container>
        <Form onSubmit={getSomething}>
          <Row>
            <Col>
          <Form.Group controlId='participants' >
            <Form.Label>Participants</Form.Label>
            <Form.Control type='text' name="totalParticipants" placeholder='Leave empty if you dont care...'></Form.Control>
          </Form.Group>
          </Col>
            <Col>
          <Form.Group controlId='priceRangeId'>
            <Form.Label>Price range</Form.Label>
            <Form.Select name="priceRange" >
              <option value="" >Select one or leave empty if you dont care</option>
              <option value="0.0">Free</option>
              <option value="0.1_0.5">Cheap</option>
              <option value="0.6_1.0">Expensive</option>
            </Form.Select>
          </Form.Group>
          </Col>
          </Row>
          <Row className='m-3'>
            <Col>
            <Form.Group>
              <Button variant="primary" type="submit">Get me something!</Button>
            </Form.Group>
            </Col>
          </Row>
        </Form>
      </Container>
    )
}

export default SearchForm
```

لاحظ دالة `getSomething`، فهناك يحدث معظم السحر. تُستدعى الدالة عند إطلاق حدث الإرسال (submit) الخاص بالنموذج.
عندما يحدث ذلك، تحصل الدالة على الحدث الاصطناعي (synthetic event) مع النموذج المرتبط داخل الخاصية `target`. نحن ببساطة نلتقط القيم من كل واحد من عوامل التصفية (حقل الإدخال والقائمة المنسدلة) ثم ننفّذ الطلب باستخدام دالة `fetch`.
لاحظ أن عنوان الـ URL لا يستهدف نقطة النهاية (endpoint) الخاصة بـ BoredAPI مباشرة. وذلك لأنه، حتى يعمل الطلب ولا يُحجب بسبب قيود CORS، قمتُ بتهيئة بروكسي (proxy) على الواجهة الخلفية لإعادة توجيه جميع الطلبات من `/api` إلى الـ API الفعلية.

والآن بعد أن رأيت الكود، فلنلقِ نظرة على ما كنت ستحصل عليه لو ثبّت أداة التتبع (tracker) الخاصة بـ OpenReplay من دون إضافة fetch.

هذه هي واجهة المستخدم لإعادة تشغيل الجلسة افتراضيًّا.
لاحظ كيف أنني في النصف السفلي قد حدّدتُ بالفعل علامة التبويب "Network"، لكن رغم أنها تُظهر الطلبات التي يجري إرسالها، فإنه لا توجد تفاصيل
عنها. وحتى إذا نقرت على أحدها، فستحصل على الحد الأدنى من التفاصيل المتاحة:

فلننظر في ذلك.

### التقاط بيانات الطلبات في عمليات إعادة تشغيل الجلسات لديك

من أجل هذا، كل ما علينا فعله هو إضافة خيار تهيئة عند إنشاء نسخة (instantiate) من أداة التتبع.
لذا الآن، عندما تكتب سطر `new tracker(...)`، ستضيف خاصية جديدة:

```javascript
import Tracker from '@openreplay/tracker';

const tracker = new Tracker({
  projectKey: "<your project key>",
  network: {
    capturePayload: true //start capturing the payload of every request
  }
});
```

هذا كل ما نحتاج إلى فعله، ومن الآن فصاعدًا، في كل مرة تنفّذ فيها طلبًا، سيتم تسجيل البيانات بواسطة أداة التتبع.
والآن انشر التغيير، واختبر التطبيق، وأغلق علامة التبويب وانتظر بضع دقائق.
من المفترض أن تظهر الجلسة قريبًا بما فيه الكفاية ويمكنك الضغط على زر "play".

## فحص الاتصال بين العميل والخادم

لأغراض هذا المثال، فلننظر أيضًا في مشكلة بدأتُ ألاحظها بعد نشر التطبيق.

لاحظ صندوق التحذير الذي أحصل عليه في هذه الحالة:

![](https://i.imgur.com/nOlRjO9.png)

بصفتي المطوّر الذي برمج هذا، أعرف ما الذي ينبغي فعله لاختباره وفهم أين يكمن الخلل (bug). ومع ذلك، بصفتي مستخدمًا، فإن الخطأ لا يخبرني حقًّا بالكثير، وقد لا أكون قادرًا على إيصال هذا بطريقة يستطيع فريق التطوير فهمها.
لذا بدلًا من ذلك، يمكنني بصفتي مستخدمًا أن أشتكي ببساطة إلى الشركة من أن تطبيقهم لا يعمل، ويمكنك أنت، بصفتك المطوّر المسؤول عن التطبيق، أن تلقي نظرة على جلستي وتفحص الطلب الذي أرسله العميل والاستجابة الواردة من الخادم.

![](/tutorials//fetch/network-tab-requests.png)

انظر الآن إلى واجهة إعادة تشغيل الجلسة. داخل علامة التبويب Network يمكنك رؤية الطلبات التي كنا نرسلها إلى الـ API الخارجية.

كل ما علينا فعله الآن هو إيجاد اللحظة التي نحصل فيها على استجابة الخطأ والنظر في الطلبات التي يجري إرسالها.
على الأرجح، سترى المشكلة داخل تفاصيل الطلب.
في حالتنا، يقول الخطأ "Failed to query due to error in arguments"، ما يعني أنه عند اختيارنا للخيار "Free" في القائمة المنسدلة،
فإننا لا نرسل طلبًا صالحًا. فلنلقِ نظرة إذًا على تفاصيله.

![](https://i.imgur.com/FAtZEgt.png)

هل ترى المشكلة؟ دعني أساعدك:

![](https://i.imgur.com/yX8TJpp.png)

نعم، إنني أرسل `undefined` كقيمة للخاصية `maxprice`. لقد أغفلتُ ذلك تمامًا في المنطق الخاص بي، والتقطتُه أثناء فحص الطلب.
صحيح أنه إصلاح سهل الآن بعد أن عرفتُ أين تكمن المشكلة، لكن بفضل هذه العملية كنت سأتمكن إما من رفع تقرير خطأ مفصّل جدًّا، أو من مساعدة المطوّر مباشرة على تحديد المشكلة وحلها من دون الاضطرار إلى اختبارها بنفسي وإعادة إنتاج المشكلة.

## وضع الخصوصية على المحك

حسنًا، فلنوسّع هذا المثال قليلًا، ولنفترض أنني أحتاج أيضًا إلى رقم هاتف مستخدمي من أجل هذا الطلب. من الواضح أنني لست بحاجة إليه، لكن جارِني للحظة.

سأضيف الحقل إلى النموذج، وسأحدّث الكود لالتقاط تلك القيمة وإرسالها كجزء من الطلب.

كود HTML الخاص بالنموذج هو مجرد إضافة عنصر `Col` جديد على النحو التالي:

```html
<!-- previous code -->
<Col>
    <Form.Group controlId='phoneNumber'>
        <Form.Label>Phone Number</Form.Label>
        <Form.Control type='number' name="phoneNumber" placeholder='Enter your phone number here please'></Form.Control>
    </Form.Group>
</Col>
<!-- rest of the code -->
```

وإضافة محتوى هذا الحقل إلى الطلب الفعلي لا تحتاج سوى سطر واحد من الكود:

```javascript
getParams.phonenumber = form.phoneNumber.value
```

والآن، ماذا يحدث إذا استخدمنا هذا الكود الجديد والتقطنا جلسة باستخدام OpenReplay؟
حسنًا، أمران:

1. إعادة التشغيل الفعلية التي _تشاهدها_ ستنقّي تلقائيًّا محتوى حقل رقم الهاتف ولن يُعرَض لأي شخص يشاهدها.
2. أما معلومات الطلب التي تلتقطها الإضافة فستُظهر القيمة، مع ذلك.

تُظهر لقطة الشاشة التالية ما وصفتُه للتو:

![](https://i.imgur.com/kKL47Rd.png)

في القسم الأيمن من الشاشة، يمكنك رؤية رقم الهاتف الكامل. يحدث هذا لأنه في حين يستطيع المتتبّع العادي أن يفهم أن حقل رقم الهاتف هو حقل رقمي، فإنه لن يلتقط ما يُدخَل فيه تحسبًّا لاحتمال أن يمثّل الرقم معلومات شخصية. لكن من جانب الطلب، لا يمكننا حقًّا افتراض ذلك لأن المطوّر قد يكون فعل أي شيء بالبيانات، أو حتى باسم المعطى (parameter).
فالسؤال إذًا هو: هل يمكننا حماية خصوصية مستخدمنا باستخدام هذه الإضافة؟

والجواب، ويسعدني أن أبلّغ به، هو: نعم نستطيع.

### تنقية بيانات الطلب

إذا عدت إلى بداية هذا الدرس، عندما هيّأتُ خيارات الشبكة (network)، فستلاحظ أنني لم أقل أي شيء عن التنقية.
ومع ذلك، كجزء من تلك الخيارات يمكنك تحديد دالة استدعاء راجعة (callback) معدّة لتنقية البيانات. تتلقى دالة الاستدعاء الراجعة هذه خاصية واحدة تحتوي على كليهما، كائن الطلب وكائن الاستجابة. يمكنك بعد ذلك أن تختار تعديلهما كيفما تشاء، فهما لن يؤثرا في الطلب الفعلي لكنهما سيغيّران الطريقة التي تُعرَض بها البيانات في واجهة OpenReplay.

على سبيل المثال، لنقل إنني أريد تغيير الخاصية "phonenumber" وإزالة الأرقام كي نتجنّب تسريب تلك المعلومات. يمكن القيام بذلك على النحو التالي:

```javascript
const tracker = new Tracker({
  projectKey: "<your project id>",
  network: {
    capturePayload: true,
    sanitizer: (data) => { //we change the content of the "phonenumber" parameter from the url
      data.url = data.url.replace(/phonenumber=([0-9]+)/, "phonenumber=XXXXXX")
      return data
    }
    }
});
```

كما ترى، التغيير بسيط، فنحن نستبدل الأرقام في هذه الخاصية فقط، بحيث يبدو الطلب الآن هكذا في واجهتنا:

![](/tutorials//fetch/network-tab-sanitized-data.png)

والآن أصبحت بيانات مستخدمك محمية من جديد.

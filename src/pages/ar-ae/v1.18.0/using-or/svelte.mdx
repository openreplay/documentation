والمفتاح لجعله يعمل هو استخدام السياق (context)، والذي، رغم أنه يبدو مشابهًا للطريقة التي نتبعها مع Next.js، إلا أن طريقة تنفيذه ستكون مختلفة.

## SvelteKit وسلوك SSR

نظرًا لأن الـ tracker الخاص بنا يتطلب سياق المتصفح لكي يعمل، فقد تحتاج إلى تعديل مثال "إنشاء سياق الـ tracker" كما هو موضح أدناه للتأكد من استيراده وإنشائه في جانب العميل (client side):

```jsx
<script>
  import { onMount } from 'svelte';

  onMount(async () => {
    if (process.browser) {
      const Tracker = await import('@openreplay/tracker');
      const tracker = new Tracker({
	      projectKey: "", // pKey
	      ingestPoint: "ingestPoint",
      })

      tracker.start()
    }
  })
</script>
```

## استخدام السياق (Context) لمشاركة البيانات

سيستخدم هذا الحل سياق Svelte (Context)، الذي يتيح لك مشاركة البيانات بين المكوّن الأب وجميع المكوّنات الأبناء التابعة له (مثل مزوّدي السياق Context في React إلى حد كبير).

في هذا المثال تحديدًا، سنضيف الـ tracker لقسم الإدارة (Admin) من تطبيقنا، لذا سنقوم بتعيين السياق داخل ملف تخطيط الإدارة (`src/layout/Admin.svelte`). ثم سنبدأ تشغيله في أول صفحة يتم عرضها داخل هذا القسم.

## إنشاء سياق الـ tracker

داخل ملف `src/layout/Admin.svelte`، وداخل وسم `script`، لنُضِف الكود التالي:

```jsx
import {Tracker, key} from "../context/tracker"

let tracker = new Tracker({
    projectKey: __myapp.env.OPENREPLAY_PROJECT_KEY
});

setContext(key, {
 getTracker: () => tracker
} )
```

بالطبع، سيكون لديك المزيد من الكود داخل هذا الملف، لكن ما تراه أعلاه هو ما يهمّنا الآن.

1. نقوم أولًا باستيراد الـ Tracker من OpenReplay ومتغيّر key (المزيد عن هذين الأمرين بعد لحظة).
2. ثم نقوم بإنشاء نسخة (instance) من الـ Tracker. وكما ترى، فإن خيار التهيئة الوحيد الذي نستخدمه هو مفتاح المشروع، والذي ستحصل عليه من الواجهة بعد إعداد المشروع في OpenReplay.
3. نقوم بحفظ نسخة الـ tracker في السياق الذي تم إنشاؤه عند استدعاء `setContext`. وسيُستخدم المتغيّر `key` لاحقًا لاسترجاع هذه القيمة المحددة من السياق.

💡**ملاحظة لمستخدمي self-hosted:** إذا كنت تستخدم النسخة المستضافة ذاتيًا (self-hosted) من OpenReplay، فسيتعين عليك أيضًا تحديد خاصية التهيئة `ingestPoint` في الخطوة 2. تحدّد هذه الخاصية نقطة نهاية الاستيعاب (ingestion endpoint) لبيانات الـ tracker. لا يحتاج مستخدمو Cloud إلى ذلك، لأن الـ tracker سيعرف افتراضيًا أين توجد نسخة SaaS من نقطة النهاية هذه، ولكن إذا
كنت تستضيفه ذاتيًا، فسيتعين عليك تحديدها (ينبغي أن تكون شيئًا مثل `https://openreplay.mydomain.com/ingest`).

## استخدام سياق الـ tracker

إن استخدام السياق (واستدعاء الدالة `start`) بسيط بقدر بساطة الحصول على السياق من خلال الدالة `getContext` من Svelte.

كان السياق يحتوي على كائن (object) به دالة تُسمى getTracker، لذا سنقوم بتفكيكه (destructure) واستدعاء تلك الدالة.

```jsx
import {key} from "../../context/tracker"

let {getTracker} = getContext(key)
onMount( () => {
	let tracker = getTracker()
	tracker.start()
})
```

بعد ذلك، يصبح الأمر مجرد استدعاء للدالة `start` من داخل دالة رد النداء (callback) الخاصة بـ `onMount`. نفعل ذلك للتأكد من أننا نستدعي هذه الدالة من المتصفح، وإلا فلن يعمل الـ tracker.

## ملف context/tracker

لقد كنا نستخدم هذا الملف في كلا الموضعين، لذا حان الوقت لإلقاء نظرة سريعة عليه.

يقوم هذا الملف بتصدير عنصرين رئيسيين:

1. الصنف (class) `Tracker`، الذي أنشأنا منه نسخة في ملف التخطيط (layout).
2. نستخدم المتغيّر `key` لحفظ السياق واسترجاعه.

وفيما يلي محتوى الملف:

```jsx
import Tracker from '@openreplay/tracker'

const key = Symbol("openreplay tracker")

export {Tracker, key}
```

الأمر المثير للاهتمام الوحيد هنا هو أن المتغيّر `key` ليس سلسلة نصية بسيطة، بل هو في الواقع `Symbol`. والسبب في ذلك هو تجنّب التعارضات المحتملة في الأسماء مع مكتبات الطرف الثالث الأخرى التي تستخدم السياق أيضًا.

## التعامل مع متغيّرات البيئة (ENV)

آخر شيء سنراجعه هنا هو كيفية تعاملنا مع مفتاح المشروع، إذ لا ينبغي أن يكون مكتوبًا بشكل ثابت (hardcoded) داخل الكود الخاص بك.

عادةً، ستستخدم متغيّر بيئة لهذا الغرض، ومع ذلك، نظرًا لأننا نحتاج إلى تلك البيانات في المتصفح ولا توفّر Svelte طريقة سهلة لتحقيق ذلك، فسنحلّ هذه المشكلة باستخدام حزمتين: [dotevn](https://www.npmjs.com/package/dotenv) و[@rollup/plugin-replace](https://www.npmjs.com/package/@rollup/plugin-replace).

من جهة، سنستخدم حزمة dotenv لنحصل على القدرة على تعريف متغيّرات البيئة الخاصة بنا داخل ملف `.env`. سنعرّف المتغيّر الذي نحتاجه:

```jsx
OPENREPLAY_PROJECT_KEY="<YOUR PROJECT KEY>"
```

من جهة أخرى، وبما أنه لا يمكننا استخدام `dotenv` في كود الواجهة الأمامية (front-end) لدينا (لأنه يعتمد على الوحدة `fs`، التي من الواضح أنها غير متوفّرة في المتصفح)، فسنستخدم إضافة replace الخاصة بـ roll-up. ستقوم هذه الإضافة باستبدال الكود في ملفاتنا بأي شيء نخبرها به. لذا سنقوم بتهيئة الإضافة لاستبدال السلسلة النصية `“__myapp.env.OPENREPLAY_PROJECT_KEY”` بمفتاح المشروع الفعلي المُحمَّل من ملف `.env` (بفضل حزمة dotenv).

ولإنجاز ذلك، افتح ملف `rollup.config.js` في جذر المشروع وأضف الكود التالي إلى مفتاح `plugins`.

```jsx
replace({
  "__myapp.env.OPENREPLAY_PROJECT_KEY": `"${process.env.OPENREPLAY_PROJECT_KEY}"`
})
```

ستعمل هذه الإضافة في الخلفية (backend) أثناء عملية التجميع (bundling)، مما يجعل عمل هذا الأمر ممكنًا. إذا كنت ترغب في فحص المحتوى الكامل لملف التهيئة هذا، يمكنك [الاطّلاع عليه هنا](https://github.com/deleteman/openreplay-svelte-example/blob/main/rollup.config.js).

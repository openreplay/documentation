يُعدّ تثبيت متتبّع OpenReplay في مشروع قائم على Gatsby أمرًا بسيطًا نسبيًا.

![Sesion replay of a Gatsby site](/images/gatsby-ss.png)

نظرًا لأن Gatsby يستخدم React في الأساس، فإن كل ما علينا فعله هو إضافة شيفرة المتتبّع داخل الصفحة الرئيسية واستدعاء الطريقة `start` باستخدام الخطّاف `useEffect`.

لنلقِ نظرة.

## لماذا نُعدّ المتتبّع على موقع ثابت؟

غياب المحتوى الديناميكي داخل تطبيقك لا يعني أنك لا ترغب في الاستفادة من بعض ميزات OpenReplay الأخرى.

ففي حين أنك لن تستخدم عمليات إعادة تشغيل الجلسات للعثور على الأخطاء، إلا أنه سيكون بإمكانك التقاط مقاييس مثيرة للاهتمام وإنشاء [لوحات معلومات مخصّصة خاصة بك](https://docs.openreplay.com/tutorials/custom-dashboard) حول هذه المقاييس.

يمكنك استخدام مقاييس الأداء و«web vitals» وحتى المقاييس المتعلقة بالموارد للحصول على فكرة عامة عن تجربة مستخدميك مع موقعك.

## إعداد المتتبّع داخل Gatsby

بافتراض وجود موقع Gatsby ثابت تكون نقطة الدخول إليه هي الملف `index.tsx` داخل مجلد `pages`، فستحتاج إلى إضافة الشيفرة التالية إليه:

```jsx
import Tracker from '@openreplay/tracker';
const tracker = new Tracker({
  projectKey: process.env.GATSBY_OPENREPLAY_PROJECT_KEY
});

const Index = ({data, location }) => {

  React.useEffect(() => {
    tracker.start();
  }, [])

   //the rest of your code here...
}
```

يتم إنشاء نسخة من المتتبّع عند تجميع الصفحة (سواء كان ذلك عبر SSR أو التصيير من جانب العميل)، لكن لا يمكن استدعاء الطريقة start إلا من المتصفح، لذا علينا التأكد من أنها تُستدعى بعد تركيب المكوّن (ومن هنا جاء استخدام الخطّاف `useEffect` هنا). إن المصفوفة الفارغة كوسيطة ثانية للخطّاف `useEffect` تضمن تنفيذ دالة الاستدعاء (callback) مرة واحدة فقط.

💡**ملاحظة لمستخدمي الاستضافة الذاتية (self-hosted):** إذا كنت تستخدم النسخة ذاتية الاستضافة من OpenReplay، فسيتعيّن عليك أيضًا تحديد خاصية الإعداد `ingestPoint` عند إنشاء نسخة من المتتبّع. تحدّد هذه الخاصية نقطة نهاية الاستيعاب (ingestion endpoint) لبيانات المتتبّع. لا يحتاج مستخدمو السحابة إلى ذلك، لأن المتتبّع سيعرف افتراضيًا أين توجد نسخة SaaS من نقطة النهاية هذه، أما إذا كنت تستضيفه ذاتيًا، فستحتاج إلى تحديدها (ينبغي أن تكون بصيغة مثل `https://openreplay.mydomain.com/ingest`)

## التعامل مع متغير ENV في Gatsby

كإجراء أمني، نوصي بعدم تضمين مفتاح المشروع بشكل ثابت (hardcoding) داخل شيفرتك. وهذا يعني أنه سيتعيّن عليك تصديره كمتغير ENV حتى يتمكّن Gatsby من قراءته.

عندها سيوجد مفتاح مشروع OpenReplay داخل متغير باسم `GATSBY_OPENREPLAY_PROJECT_KEY`

لاحظ كيف أن الاسم يبدأ بالبادئة `GATSBY_`، وهو ما سيُخبر Gatsby بأن يتأكّد من توفّر المتغير في شيفرة الواجهة الأمامية (front-end) أيضًا. وإلا فلن تتمكّن من قراءة القيمة ما لم تكن تُنفّذ شيفرة Node.js.

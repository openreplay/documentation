تثبيت أداة التتبّع (tracker) الخاصة بـ OpenReplay في مشروع مبني على Svelte أمرٌ بسيط نسبياً.

المفتاح لجعلها تعمل هو استخدام السياق (context)، والذي، رغم أنه يبدو مشابهاً للطريقة التي نتّبعها مع Next.js، إلا أن طريقة التنفيذ ستكون مختلفة.

## SvelteKit وسلوك SSR

بما أن أداة التتبّع لدينا تتطلب سياق المتصفح كي تعمل، فقد تحتاج إلى تعديل مثال "إنشاء سياق أداة التتبّع" كما هو موضّح أدناه لضمان استيرادها وإنشائها في جانب العميل (client side):

```jsx
<script>
  import { onMount } from 'svelte';

  onMount(async () => {
    if (process.browser) {
      const Tracker = await import('@openreplay/tracker');
      const tracker = new Tracker({
	      projectKey: "", // pKey
	      ingestPoint: "ingestPoint",
      })

      tracker.start()
    }
  })
</script>
```

## استخدام السياق (Context) لمشاركة البيانات

سيستخدم هذا الحل ميزة Context الخاصة بـ Svelte، والتي تتيح لك مشاركة البيانات بين المكوّن الأب وجميع المكوّنات الأبناء التابعة له (إلى حدّ كبير مثل موفّري Context في React).

في هذا المثال على وجه التحديد، سنضيف أداة التتبّع لقسم الإدارة (Admin) في تطبيقنا، لذا سنقوم بتعيين السياق داخل ملف التخطيط (layout) الخاص بالإدارة (`src/layout/Admin.svelte`). ثم سنقوم بتشغيلها في أول صفحة يتم عرضها داخل هذا القسم.

## إنشاء سياق أداة التتبّع

داخل ملف `src/layout/Admin.svelte`، وضمن وسم `script`، لنُضِف الكود التالي:

```jsx
import {Tracker, key} from "../context/tracker"

let tracker = new Tracker({
    projectKey: __myapp.env.OPENREPLAY_PROJECT_KEY
});

setContext(key, {
 getTracker: () => tracker
} )
```

بالطبع، سيكون لديك المزيد من الكود داخل هذا الملف، لكن ما تراه أعلاه هو ما يهمّنا في الوقت الحالي.

1. نقوم أولاً باستيراد Tracker من OpenReplay ومتغيّر key (سنتحدث عن هذه الأمور بعد قليل).
2. ثم نقوم بإنشاء نسخة (instance) من Tracker. كما ترى، خيار التهيئة الوحيد الذي نستخدمه هو مفتاح المشروع (project key)، والذي يمكنك الحصول عليه من واجهة المستخدم بعد إعداد المشروع في OpenReplay.
3. نقوم بحفظ نسخة أداة التتبّع في السياق الذي تم إنشاؤه عبر استدعاء `setContext`. سيُستخدم المتغيّر `key` لاحقاً لاسترجاع هذه القيمة المحدّدة من السياق.

💡**ملاحظة لمستخدمي النسخة المستضافة ذاتياً (self-hosted):** إذا كنت تستخدم النسخة المستضافة ذاتياً من OpenReplay، فسيتعيّن عليك أيضاً تحديد خاصية التهيئة `ingestPoint` في الخطوة 2. تحدّد هذه الخاصية نقطة استقبال (ingestion endpoint) بيانات أداة التتبّع. لا يحتاج مستخدمو السحابة (Cloud) إلى ذلك، لأن أداة التتبّع، افتراضياً، ستعرف أين توجد نسخة SaaS من نقطة النهاية هذه، لكن إذا
كنت تستضيفها ذاتياً، فستحتاج إلى تحديدها (ينبغي أن تكون شيئاً مثل `https://openreplay.mydomain.com/ingest`).

## استخدام سياق أداة التتبّع

استخدام السياق (واستدعاء الدالة `start`) بسيط بقدر الحصول على السياق عبر الدالة `getContext` من Svelte.

كان السياق يحتوي على كائن يضمّ دالة تُسمى getTracker، لذا سنقوم بتفكيكها (destructure) واستدعاء تلك الدالة.

```jsx
import {key} from "../../context/tracker"

let {getTracker} = getContext(key)
onMount( () => {
	let tracker = getTracker()
	tracker.start()
})
```

بعد ذلك، يكفي استدعاء الدالة `start` من داخل ردّ النداء (callback) الخاص بـ `onMount`. نفعل ذلك لضمان أننا نستدعي هذه الدالة من المتصفح، وإلا فلن تعمل أداة التتبّع.

## ملف context/tracker

لقد كنا نستخدم هذا الملف في كلا الموضعين، لذا حان الوقت لإلقاء نظرة سريعة عليه.

يقوم هذا الملف بتصدير عنصرين رئيسيين:

1. الصنف (class) `Tracker`، الذي أنشأنا نسخة منه في ملف التخطيط (layout).
2. نستخدم المتغيّر `key` لحفظ السياق واسترجاعه.

إليك محتوى الملف:

```jsx
import Tracker from '@openreplay/tracker'

const key = Symbol("openreplay tracker")

export {Tracker, key}
```

الأمر الوحيد المثير للاهتمام هنا هو أن المتغيّر `key` ليس مجرد سلسلة نصية بسيطة، بل هو في الواقع `Symbol`. والسبب في ذلك هو تجنّب التعارضات المحتملة في الأسماء مع مكتبات الطرف الثالث الأخرى التي تستخدم السياق أيضاً.

## التعامل مع متغيّرات البيئة (ENV)

آخر ما سنراجعه هنا هو كيفية تعاملنا مع مفتاح المشروع (project key)، إذ لا ينبغي ترميزه بشكل ثابت (hardcoded) داخل الكود الخاص بك.

عادةً، ستستخدم متغيّر بيئة لهذا الغرض، ولكن بما أننا نحتاج إلى تلك البيانات في المتصفح وأن Svelte لا توفّر طريقة سهلة لتحقيق ذلك، سنحلّ هذه المشكلة باستخدام حزمتين: [dotevn](https://www.npmjs.com/package/dotenv) و[@rollup/plugin-replace](https://www.npmjs.com/package/@rollup/plugin-replace).

من جهة، سنستخدم حزمة dotenv لاكتساب القدرة على تعريف متغيّرات البيئة الخاصة بنا داخل ملف `.env`. سنعرّف المتغيّر الذي نحتاجه:

```jsx
OPENREPLAY_PROJECT_KEY="<YOUR PROJECT KEY>"
```

ومن جهة أخرى، بما أننا لا نستطيع استخدام `dotenv` في كود الواجهة الأمامية (front-end) لدينا (لأنه يعتمد على الوحدة `fs`، والتي من الواضح أنها غير متوفرة في المتصفح)، سنستخدم إضافة replace الخاصة بـ roll-up. ستقوم هذه الإضافة باستبدال الكود في ملفاتنا بأي شيء نحدّده لها. لذا سنقوم بتهيئة الإضافة لاستبدال السلسلة النصية `“__myapp.env.OPENREPLAY_PROJECT_KEY”` بمفتاح المشروع الفعلي المحمّل من ملف `.env` (بفضل حزمة dotenv).

لإنجاز ذلك، افتح ملف `rollup.config.js` الموجود في جذر المشروع وأضِف الكود التالي إلى المفتاح `plugins`.

```jsx
replace({
  "__myapp.env.OPENREPLAY_PROJECT_KEY": `"${process.env.OPENREPLAY_PROJECT_KEY}"`
})
```

ستعمل هذه الإضافة في الخلفية (backend) أثناء عملية التجميع (bundling)، مما يجعل عملها ممكناً. إذا كنت ترغب في فحص المحتوى الكامل لملف التهيئة هذا، يمكنك [الاطّلاع عليه هنا](https://github.com/deleteman/openreplay-svelte-example/blob/main/rollup.config.js).

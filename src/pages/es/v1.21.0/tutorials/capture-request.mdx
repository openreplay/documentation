A veces los errores en tu código de cliente no son tan evidentes como cuando obtienes una pantalla en blanco porque nada de tu código JavaScript funciona.
A veces el problema con tu aplicación es que, ante ciertas acciones que tu usuario puede realizar, estás formando incorrectamente una solicitud al servidor.
El código de cliente funciona, no obtienes ningún error de JS en la consola, pero a tu back-end realmente no le gusta lo que le estás enviando.

Con OpenReplay, puedes capturar la comunicación cliente-servidor como parte de tu repetición de sesión estándar y revisarla más tarde. Así que veamos cómo podemos hacerlo y qué tipo de beneficio podemos obtener de ello.

## Una aplicación de ejemplo
Para los propósitos de este tutorial, he creado una sencilla aplicación de React que hace uso de la [Bored API](https://www.boredapi.com/). Esta es una API muy sencilla que devuelve una sugerencia de actividad aleatoria basada en algunos parámetros.
Así que creé la "I'm bored App", que se ve así:

![](https://i.imgur.com/ZCEOtiC.png)

Y puedes encontrarla en vivo en Netlify [aquí](https://venerable-tartufo-f87748.netlify.app), o si quieres revisar el código para verlo en detalle, está totalmente disponible en [GitHub](https://github.com/deleteman/im-bored-app).

Esta aplicación está formada por 2 componentes, el componente `SearchForm` que se encarga de renderizar esos 2 campos y el botón, así como de enviar la solicitud real a la API.
Y el componente `Suggestion` simplemente renderiza la sugerencia dentro de una caja de aspecto agradable.

Voy a centrarme en el primero, ya que es el único que envía solicitudes usando la función `fetch`.

**Ten en cuenta** que esta técnica también funciona para cualquier solicitud realizada con **Axios**.

Echemos un vistazo rápido al componente para entender qué está haciendo.

### El código del componente SearchForm

Este no es un componente complejo, pero hay una sección que es especialmente relevante para este caso de uso particular, así que echémosle un vistazo rápido.

```javascript
import { Container, Col, Form, Row, Button } from 'react-bootstrap';

const SearchForm = ({setResult, fetcher}) => {
const getSomething = async (evt) => {
    evt.preventDefault()
    let form = evt.target

    const API_URL = "/api/activity?"
    let getParams = {}
    if(form.participants.value !== '') {
      getParams.participants = form.participants.value
    }

    if(form.priceRange.value !== '') {
      let prices = form.priceRange.value.split("_")
      getParams.minprice = prices[0]
      getParams.maxprice = prices[1]
    }

    let results = await fetcher(API_URL + new URLSearchParams(getParams), {
        mode: 'no-cors'
    })
    setResult(await results.json())

    return false
  }

    return (
        <Container>
        <Form onSubmit={getSomething}>
          <Row>
            <Col>
          <Form.Group controlId='participants' >
            <Form.Label>Participants</Form.Label>
            <Form.Control type='text' name="totalParticipants" placeholder='Leave empty if you dont care...'></Form.Control>
          </Form.Group>
          </Col>
            <Col>
          <Form.Group controlId='priceRangeId'>
            <Form.Label>Price range</Form.Label>
            <Form.Select name="priceRange" >
              <option value="" >Select one or leave empty if you dont care</option>
              <option value="0.0">Free</option>
              <option value="0.1_0.5">Cheap</option>
              <option value="0.6_1.0">Expensive</option>
            </Form.Select>
          </Form.Group>
          </Col>
          </Row>
          <Row className='m-3'>
            <Col>
            <Form.Group>
              <Button variant="primary" type="submit">Get me something!</Button>
            </Form.Group>
            </Col>
          </Row>
        </Form>
      </Container>
    )
}

export default SearchForm
```

Fíjate en la función `getSomething`, ahí es donde ocurre la mayor parte de la magia. La función se llama cuando se desencadena el evento de envío del formulario.
Cuando eso sucede, la función obtiene el evento sintético con el formulario vinculado dentro de la propiedad `target`. Simplemente capturamos los valores de cada uno de los filtros (el campo de entrada y el menú desplegable) y luego ejecutamos la solicitud con la función `fetch`.
Fíjate en que la URL no apunta directamente al endpoint de la BoredAPI. Eso es porque, para que la solicitud funcione y no quede bloqueada debido a las restricciones de CORS, he configurado un proxy en el backend para redirigir todas las solicitudes de `/api` a la API real.

Ahora que has visto el código, echemos un vistazo a lo que obtendrías si instalaras el tracker de OpenReplay sin el plugin de fetch.

### Captura de los datos de solicitudes en tus repeticiones de sesión

Para esto, todo lo que tenemos que hacer es añadir una opción de configuración cuando instancias el tracker.
Así que ahora, cuando escribas la línea `new tracker(...)`, añadirás una nueva propiedad:

```javascript
import Tracker from '@openreplay/tracker';

const tracker = new Tracker({
  projectKey: "<your project key>",
  network: {
    capturePayload: true //start capturing the payload of every request
  }
});
```

Eso es todo lo que necesitamos hacer; de ahora en adelante, cada vez que realices una solicitud, los datos serán registrados por el tracker.
Ahora despliega el cambio, prueba la aplicación, cierra la pestaña y espera un par de minutos.
La sesión debería aparecer pronto y puedes pulsar el botón "play".

## Poniendo la privacidad a prueba

Muy bien, llevemos este ejemplo un poco más allá; supongamos que también necesito el número de teléfono de mi usuario para esta solicitud. Claramente no lo necesito, pero sígueme la corriente por un minuto.

Añadiré el campo al formulario, y actualizaré el código para capturar ese valor y enviarlo como parte de la solicitud.

El HTML del formulario consiste solo en añadir un nuevo elemento `Col` así:

```html
<!-- previous code -->
<Col>
    <Form.Group controlId='phoneNumber'>
        <Form.Label>Phone Number</Form.Label>
        <Form.Control type='number' name="phoneNumber" placeholder='Enter your phone number here please'></Form.Control>
    </Form.Group>
</Col>
<!-- rest of the code -->
```

Y añadir el contenido de este campo a la solicitud real solo necesita una única línea de código:

```javascript
getParams.phonenumber = form.phoneNumber.value
```

Ahora, ¿qué pasa si usamos este nuevo código y capturamos una sesión con OpenReplay?
Bueno, dos cosas:

1. La repetición que realmente _ves_ saneará automáticamente el contenido del campo del número de teléfono y no se le mostrará a nadie que la esté viendo.
2. La información de la solicitud capturada por el plugin, sin embargo, mostrará el valor.

La siguiente captura de pantalla muestra lo que acabo de describir:

![](https://i.imgur.com/kKL47Rd.png)

En la sección derecha de la pantalla, puedes ver el número de teléfono completo. Esto sucede porque, mientras que el tracker normal puede entender el campo del número de teléfono como un campo numérico, no capturará su entrada por si acaso el número representa información personal. Pero en el lado de la solicitud, realmente no podemos hacer esa suposición ya que el desarrollador podría haber hecho cualquier cosa con los datos, o incluso con el nombre del parámetro.
Así que la pregunta entonces es: ¿podemos proteger la privacidad de nuestro usuario con este plugin?

Y la respuesta, me complace informar, es: SÍ podemos.

### Saneamiento de los datos de la solicitud

Si vuelves al principio de este tutorial, cuando configuré las opciones de network, verás que no dije nada sobre el saneamiento.
Sin embargo, como parte de esas opciones puedes especificar un callback destinado a sanear datos. Este callback recibe un único atributo con ambos, el objeto de la solicitud y el de la respuesta. Luego puedes elegir editarlos como quieras; no afectarán a la solicitud real, pero cambiarán la manera en que se muestran los datos en la interfaz de OpenReplay.

Por ejemplo, supongamos que quiero cambiar el atributo "phonenumber" y eliminar los números para evitar filtrar esa información. Esto se puede hacer así:

```javascript
const tracker = new Tracker({
  projectKey: "<your project id>",
  network: {
    capturePayload: true,
    sanitizer: (data) => { //we change the content of the "phonenumber" parameter from the url
      data.url = data.url.replace(/phonenumber=([0-9]+)/, "phonenumber=XXXXXX")
      return data
    }
    }
});
```

Como puedes ver, el cambio es sencillo, reemplazamos solo los números en este atributo, así que ahora la solicitud se ve así en nuestra interfaz:

![](/tutorials//fetch/network-tab-sanitized-data.png)

Ahora los datos de tu usuario están protegidos de nuevo.

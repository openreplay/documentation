Parfois, les erreurs dans votre code client ne sont pas aussi évidentes que lorsque vous obtenez un écran blanc parce qu'aucune partie de votre code JavaScript ne fonctionne.
Parfois, le problème avec votre application est que, pour certaines actions que votre utilisateur peut effectuer, vous formez incorrectement une requête vers le serveur.
Le code client fonctionne, vous n'obtenez aucune erreur JS dans la console, mais votre back-end n'aime vraiment pas ce que vous lui envoyez.

Avec OpenReplay, vous pouvez capturer la communication client-serveur dans le cadre de votre relecture de session standard et l'examiner ultérieurement. Voyons donc comment nous pouvons faire cela et quel genre d'avantage nous pouvons en tirer.

## Une application d'exemple
Pour les besoins de ce guide pratique, j'ai créé une application React simple qui utilise la [Bored API](https://www.boredapi.com/). Il s'agit d'une API très simple qui renvoie une suggestion d'activité aléatoire en fonction de certains paramètres.
J'ai donc créé l'« I'm bored App », qui ressemble à ceci :

![](https://i.imgur.com/ZCEOtiC.png)

Et vous pouvez la trouver en ligne sur Netlify [ici](https://venerable-tartufo-f87748.netlify.app), ou si vous souhaitez examiner le code en détail, il est entièrement disponible sur [GitHub](https://github.com/deleteman/im-bored-app).

Cette application est constituée de 2 composants, le composant `SearchForm` qui se charge de rendre ces 2 champs et le bouton, ainsi que d'envoyer la requête proprement dite à l'API.
Et le composant `Suggestion` se contente de rendre la suggestion à l'intérieur d'une jolie boîte.

Je vais me concentrer sur le premier, car c'est le seul qui envoie des requêtes à l'aide de la fonction `fetch`.

**Notez** que cette technique fonctionne également pour toute requête effectuée avec **Axios**.

Jetons un coup d'œil rapide au composant pour comprendre ce qu'il fait.

### Le code du composant SearchForm

Ce n'est pas un composant complexe, mais il y a une section qui est particulièrement pertinente pour ce cas d'usage particulier, alors examinons-la rapidement.

```javascript
import { Container, Col, Form, Row, Button } from 'react-bootstrap';

const SearchForm = ({setResult, fetcher}) => {
const getSomething = async (evt) => {
    evt.preventDefault()
    let form = evt.target

    const API_URL = "/api/activity?"
    let getParams = {}
    if(form.participants.value !== '') {
      getParams.participants = form.participants.value
    }

    if(form.priceRange.value !== '') {
      let prices = form.priceRange.value.split("_")
      getParams.minprice = prices[0]
      getParams.maxprice = prices[1]
    }

    let results = await fetcher(API_URL + new URLSearchParams(getParams), {
        mode: 'no-cors'
    })
    setResult(await results.json())

    return false
  }

    return (
        <Container>
        <Form onSubmit={getSomething}>
          <Row>
            <Col>
          <Form.Group controlId='participants' >
            <Form.Label>Participants</Form.Label>
            <Form.Control type='text' name="totalParticipants" placeholder='Leave empty if you dont care...'></Form.Control>
          </Form.Group>
          </Col>
            <Col>
          <Form.Group controlId='priceRangeId'>
            <Form.Label>Price range</Form.Label>
            <Form.Select name="priceRange" >
              <option value="" >Select one or leave empty if you dont care</option>
              <option value="0.0">Free</option>
              <option value="0.1_0.5">Cheap</option>
              <option value="0.6_1.0">Expensive</option>
            </Form.Select>
          </Form.Group>
          </Col>
          </Row>
          <Row className='m-3'>
            <Col>
            <Form.Group>
              <Button variant="primary" type="submit">Get me something!</Button>
            </Form.Group>
            </Col>
          </Row>
        </Form>
      </Container>
    )
}

export default SearchForm
```

Remarquez la fonction `getSomething`, c'est là que se passe la majeure partie de la magie. La fonction est appelée lorsque l'événement de soumission du formulaire est déclenché.
Lorsque cela se produit, la fonction obtient l'événement synthétique avec le formulaire lié à l'intérieur de la propriété `target`. Nous capturons simplement les valeurs de chacun des filtres (le champ de saisie et le menu déroulant) puis nous exécutons la requête avec la fonction `fetch`.
Remarquez que l'URL ne cible pas directement le point de terminaison de la BoredAPI. C'est parce que, pour que la requête fonctionne et ne soit pas bloquée à cause des restrictions CORS, j'ai configuré un proxy sur le backend afin de rediriger toutes les requêtes de `/api` vers l'API réelle.

Maintenant que vous avez vu le code, voyons ce que vous obtiendriez si vous installiez le tracker d'OpenReplay sans le plugin fetch.

### Capturer les données de requête dans vos relectures de session

Pour cela, tout ce que nous avons à faire est d'ajouter une option de configuration lorsque vous instanciez le tracker.
Désormais, lorsque vous écrirez la ligne `new tracker(...)`, vous ajouterez une nouvelle propriété :

```javascript
import Tracker from '@openreplay/tracker';

const tracker = new Tracker({
  projectKey: "<your project key>",
  network: {
    capturePayload: true //start capturing the payload of every request
  }
});
```

C'est tout ce que nous devons faire ; à partir de maintenant, chaque fois que vous effectuez une requête, les données seront enregistrées par le tracker.
Maintenant, déployez le changement, testez l'application, fermez l'onglet et attendez quelques minutes.
La session devrait apparaître assez rapidement et vous pouvez appuyer sur le bouton « play ».

## Mettre la confidentialité à l'épreuve

Très bien, poussons cet exemple un peu plus loin ; faisons comme si j'avais aussi besoin du numéro de téléphone de mon utilisateur pour cette requête. Je n'en ai clairement pas besoin, mais jouez le jeu une minute.

Je vais ajouter le champ au formulaire, et je vais mettre à jour le code pour capturer cette valeur et l'envoyer dans le cadre de la requête.

Le code HTML du formulaire consiste simplement à ajouter un nouvel élément `Col` comme ceci :

```html
<!-- previous code -->
<Col>
    <Form.Group controlId='phoneNumber'>
        <Form.Label>Phone Number</Form.Label>
        <Form.Control type='number' name="phoneNumber" placeholder='Enter your phone number here please'></Form.Control>
    </Form.Group>
</Col>
<!-- rest of the code -->
```

Et l'ajout du contenu de ce champ dans la requête proprement dite ne nécessite qu'une seule ligne de code :

```javascript
getParams.phonenumber = form.phoneNumber.value
```

Maintenant, que se passe-t-il si nous utilisons ce nouveau code et capturons une session avec OpenReplay ?
Eh bien, deux choses :

1. La relecture que vous _regardez_ assainira automatiquement le contenu du champ du numéro de téléphone et il ne sera affiché à personne le visionnant.
2. Les informations de requête capturées par le plugin afficheront toutefois la valeur.

La capture d'écran suivante montre ce que je viens de décrire :

![](https://i.imgur.com/kKL47Rd.png)

Dans la section droite de l'écran, vous pouvez voir le numéro de téléphone complet. Cela se produit parce que, bien que le tracker normal puisse comprendre que le champ du numéro de téléphone est un champ numérique, il ne capturera pas sa saisie au cas où le numéro représenterait des informations personnelles. Mais du côté de la requête, nous ne pouvons pas vraiment faire cette hypothèse puisque le développeur aurait pu faire n'importe quoi avec les données, voire avec le nom du paramètre.
La question est donc : pouvons-nous protéger la confidentialité de notre utilisateur avec ce plugin ?

Et la réponse, je suis heureux de l'annoncer, est : OUI, nous le pouvons.

### Assainir les données de la requête

Si vous revenez au début de ce tutoriel, lorsque j'ai configuré les options réseau, vous verrez que je n'ai rien dit à propos de l'assainissement.
Cependant, dans le cadre de ces options, vous pouvez spécifier un callback destiné à assainir les données. Ce callback reçoit un seul attribut contenant à la fois l'objet de la requête et celui de la réponse. Vous pouvez alors choisir de les modifier comme bon vous semble ; cela n'affectera pas la requête réelle mais changera la façon dont les données sont affichées dans l'interface d'OpenReplay.

Par exemple, supposons que je veuille modifier l'attribut « phonenumber » et supprimer les chiffres afin d'éviter de divulguer cette information. Cela peut être fait comme ceci :

```javascript
const tracker = new Tracker({
  projectKey: "<your project id>",
  network: {
    capturePayload: true,
    sanitizer: (data) => { //we change the content of the "phonenumber" parameter from the url
      data.url = data.url.replace(/phonenumber=([0-9]+)/, "phonenumber=XXXXXX")
      return data
    }
    }
});
```

Comme vous pouvez le voir, le changement est simple, nous remplaçons uniquement les chiffres de cet attribut, de sorte que la requête ressemble désormais à ceci dans notre interface :

![](/tutorials//fetch/network-tab-sanitized-data.png)

Maintenant, les données de votre utilisateur sont à nouveau protégées.

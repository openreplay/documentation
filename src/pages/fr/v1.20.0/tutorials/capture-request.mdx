Parfois, les erreurs dans votre code client ne sont pas aussi évidentes que lorsque vous obtenez un écran blanc parce qu'aucune partie de votre code JavaScript ne fonctionne.
Parfois, le problème avec votre application est que, pour certaines actions que votre utilisateur peut effectuer, vous formez incorrectement une requête vers le serveur.
Le code client fonctionne, vous n'obtenez aucune erreur JS dans la console, mais votre back-end n'apprécie pas vraiment ce que vous lui envoyez.

Avec OpenReplay, vous pouvez capturer la communication client-serveur dans le cadre de votre replay de session standard, et la consulter plus tard. Voyons donc comment nous pouvons faire cela et quel type d'avantage nous pouvons en tirer.

## Une application d'exemple
Pour les besoins de ce tutoriel, j'ai créé une application React simple qui utilise la [Bored API](https://www.boredapi.com/). Il s'agit d'une API très simple qui renvoie une suggestion d'activité aléatoire en fonction de certains paramètres.
J'ai donc créé l'application « I'm bored App », qui ressemble à ceci :

![](https://i.imgur.com/ZCEOtiC.png)

Et vous pouvez la trouver en ligne sur Netlify [ici](https://venerable-tartufo-f87748.netlify.app), ou si vous souhaitez consulter le code pour l'examiner en détail, il est entièrement disponible sur [GitHub](https://github.com/deleteman/im-bored-app).

Cette application est composée de 2 composants, le composant `SearchForm`, qui s'occupe d'afficher ces 2 champs et le bouton, ainsi que d'envoyer la requête réelle à l'API.
Et le composant `Suggestion` se contente d'afficher la suggestion dans une jolie boîte.

Je vais me concentrer sur le premier, car c'est le seul qui envoie des requêtes en utilisant la fonction `fetch`.

**Notez** que cette technique fonctionne également pour toute requête effectuée avec **Axios**.

Jetons un coup d'œil rapide au composant pour comprendre ce qu'il fait.

### Le code du composant SearchForm

Ce n'est pas un composant complexe, mais il y a une section particulièrement pertinente pour ce cas d'usage précis, alors jetons-y un coup d'œil rapide.

```javascript
import { Container, Col, Form, Row, Button } from 'react-bootstrap';

const SearchForm = ({setResult, fetcher}) => {
const getSomething = async (evt) => {
    evt.preventDefault()
    let form = evt.target

    const API_URL = "/api/activity?"
    let getParams = {}
    if(form.participants.value !== '') {
      getParams.participants = form.participants.value
    }

    if(form.priceRange.value !== '') {
      let prices = form.priceRange.value.split("_")
      getParams.minprice = prices[0]
      getParams.maxprice = prices[1]
    }

    let results = await fetcher(API_URL + new URLSearchParams(getParams), {
        mode: 'no-cors'
    })
    setResult(await results.json())

    return false
  }

    return (
        <Container>
        <Form onSubmit={getSomething}>
          <Row>
            <Col>
          <Form.Group controlId='participants' >
            <Form.Label>Participants</Form.Label>
            <Form.Control type='text' name="totalParticipants" placeholder='Leave empty if you dont care...'></Form.Control>
          </Form.Group>
          </Col>
            <Col>
          <Form.Group controlId='priceRangeId'>
            <Form.Label>Price range</Form.Label>
            <Form.Select name="priceRange" >
              <option value="" >Select one or leave empty if you dont care</option>
              <option value="0.0">Free</option>
              <option value="0.1_0.5">Cheap</option>
              <option value="0.6_1.0">Expensive</option>
            </Form.Select>
          </Form.Group>
          </Col>
          </Row>
          <Row className='m-3'>
            <Col>
            <Form.Group>
              <Button variant="primary" type="submit">Get me something!</Button>
            </Form.Group>
            </Col>
          </Row>
        </Form>
      </Container>
    )
}

export default SearchForm
```

Remarquez la fonction `getSomething`, c'est là que la plus grande partie de la magie opère. La fonction est appelée lorsque l'événement submit du formulaire est déclenché.
Lorsque cela se produit, la fonction récupère l'événement synthétique avec le formulaire lié à l'intérieur de la propriété `target`. Nous capturons simplement les valeurs de chacun des filtres (le champ de saisie et la liste déroulante), puis nous exécutons la requête avec la fonction `fetch`.
Remarquez que l'URL ne cible pas directement l'endpoint de la BoredAPI. C'est parce que, pour que la requête fonctionne et ne soit pas bloquée à cause des restrictions CORS, j'ai configuré un proxy sur le backend pour rediriger toutes les requêtes de `/api` vers l'API réelle.

Maintenant que vous avez vu le code, jetons un coup d'œil à ce que vous obtiendriez si vous installiez le tracker d'OpenReplay sans le plugin fetch.

Examinons cela.

### Capturer les données de requête dans vos replays de session

Pour cela, tout ce que nous avons à faire est d'ajouter une option de configuration lorsque vous instanciez le tracker.
Désormais, lorsque vous écrirez la ligne `new tracker(...)`, vous ajouterez une nouvelle propriété :

```javascript
import Tracker from '@openreplay/tracker';

const tracker = new Tracker({
  projectKey: "<your project key>",
  network: {
    capturePayload: true //start capturing the payload of every request
  }
});
```

C'est tout ce que nous avons à faire ; à partir de maintenant, chaque fois que vous effectuez une requête, les données seront enregistrées par le tracker.
Déployez maintenant la modification, testez l'application, fermez l'onglet et attendez quelques minutes.
La session devrait apparaître assez vite et vous pourrez appuyer sur le bouton « play ».

Alors, à la place, en tant qu'utilisateur, je peux simplement me plaindre auprès de l'entreprise que leur application ne fonctionne pas, et vous, en tant que développeur responsable de l'application, pouvez jeter un coup d'œil à ma session et inspecter la requête envoyée par le client ainsi que la réponse du serveur.

## Mettre la confidentialité à l'épreuve

Très bien, poussons cet exemple un peu plus loin ; imaginons que j'ai également besoin du numéro de téléphone de mon utilisateur pour cette requête. Clairement, je n'en ai pas besoin, mais jouez le jeu une minute.

Je vais ajouter le champ au formulaire, et je vais mettre à jour le code pour capturer cette valeur et l'envoyer dans le cadre de la requête.

Le HTML du formulaire consiste simplement à ajouter un nouvel élément `Col` comme ceci :

```html
<!-- previous code -->
<Col>
    <Form.Group controlId='phoneNumber'>
        <Form.Label>Phone Number</Form.Label>
        <Form.Control type='number' name="phoneNumber" placeholder='Enter your phone number here please'></Form.Control>
    </Form.Group>
</Col>
<!-- rest of the code -->
```

Et ajouter le contenu de ce champ à la requête réelle ne nécessite qu'une seule ligne de code :

```javascript
getParams.phonenumber = form.phoneNumber.value
```

Maintenant, que se passe-t-il si nous utilisons ce nouveau code et capturons une session avec OpenReplay ?
Eh bien, deux choses :

1. Le replay réel que vous _regardez_ assainira automatiquement le contenu du champ du numéro de téléphone et il ne sera affiché à personne qui le visionne.
2. Les informations de requête capturées par le plugin afficheront en revanche la valeur.

La capture d'écran suivante montre ce que je viens de décrire :

![](https://i.imgur.com/kKL47Rd.png)

Dans la partie droite de l'écran, vous pouvez voir le numéro de téléphone complet. Cela se produit parce que, si le tracker normal peut comprendre que le champ du numéro de téléphone est un champ numérique, il ne capturera pas sa saisie au cas où le numéro représenterait des informations personnelles. Mais du côté de la requête, nous ne pouvons pas vraiment faire cette supposition, car le développeur aurait pu faire n'importe quoi avec les données, ou même avec le nom du paramètre.
La question est donc : pouvons-nous protéger la confidentialité de notre utilisateur avec ce plugin ?

Et la réponse, je suis heureux de l'annoncer, est : OUI, nous le pouvons.

### Assainir les données de requête

Si vous revenez au début de ce tutoriel, lorsque j'ai configuré les options de network, vous verrez que je n'ai rien dit au sujet de l'assainissement.
Cependant, dans le cadre de ces options, vous pouvez spécifier un callback destiné à assainir les données. Ce callback reçoit un seul attribut contenant à la fois l'objet de requête et l'objet de réponse. Vous pouvez alors choisir de les modifier comme bon vous semble ; ils n'affecteront pas la requête réelle mais changeront la manière dont les données sont affichées dans l'interface d'OpenReplay.

Par exemple, disons que je veux modifier l'attribut « phonenumber » et supprimer les chiffres afin d'éviter de divulguer cette information. Cela peut se faire ainsi :

```javascript
const tracker = new Tracker({
  projectKey: "<your project id>",
  network: {
    capturePayload: true,
    sanitizer: (data) => { //we change the content of the "phonenumber" parameter from the url
      data.url = data.url.replace(/phonenumber=([0-9]+)/, "phonenumber=XXXXXX")
      return data
    }
    }
});
```

Comme vous pouvez le voir, le changement est simple, nous remplaçons uniquement les chiffres de cet attribut, de sorte que la requête ressemble maintenant à ceci dans notre interface :

![](/tutorials/fetch/network-tab-sanitized-data.png)

Désormais, les données de votre utilisateur sont à nouveau protégées.

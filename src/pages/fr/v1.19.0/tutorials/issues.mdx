En plus de fournir d'excellents outils pour vous aider à déboguer votre application, OpenReplay catégorise automatiquement les différents types d'événements à l'intérieur du replay et vous indique quel genre de problèmes s'y trouvent.

C'est une excellente façon de filtrer rapidement les sessions problématiques parmi les potentiellement milliers que votre produit pourrait générer quotidiennement.

## Comprendre les différents types de problèmes

Pour faire simple, tous les types de problèmes sont listés sur le côté gauche de votre écran chaque fois que vous consultez la liste des sessions.

![](https://i.imgur.com/2Rygjwf.png)

Vous n'avez rien de particulier à faire pour qu'OpenReplay les détecte et les identifie. C'est le meilleur aspect : dès lors que vous commencez à enregistrer des sessions, vous pouvez ensuite ouvrir la liste des replays et filtrer par l'ensemble des problèmes détectés.

### Problèmes détectés par OpenReplay

Il existe au total 8 types de problèmes différents détectés par OpenReplay que vous pouvez utiliser comme filtres sur votre liste principale de replays.

1. **Erreurs** : Toute exception non gérée déclenchée par votre application sera détectée par OpenReplay et mise en évidence sur votre chronologie pendant la production.
2. **Mauvaises requêtes** : Chaque fois que votre code envoie une requête qui reçoit un code de statut 4XX, elle sera signalée sur votre chronologie.
3. **Images manquantes** : Toute image que vous référencez de manière incorrecte ou qui, pour toute autre raison, ne parvient pas à se charger, signalera le replay afin que vous puissiez comprendre ce qui manque.
4. **Click rage** : Cela se déclenche chaque fois que votre utilisateur clique plusieurs fois sur le même élément. Que ces clics produisent un résultat ou non, lorsqu'un utilisateur clique de nombreuses fois sur le même élément, c'est le signe que quelque chose pourrait ne pas aller avec votre UX. OpenReplay signale donc la session afin que vous puissiez examiner où et quand cela se produit.
5. **Dead clicks** : Ce sont des clics qui ne produisent aucun résultat. Essentiellement, chaque fois que vos utilisateurs cliquent sur un bouton, un lien ou tout autre élément de leur écran, nous le détecterons et signalerons la session. Les dead clicks sont un indicateur clair qu'il y a un problème avec votre conception, car elle amène vos utilisateurs à croire qu'ils peuvent cliquer à un endroit où ils ne le peuvent pas réellement. Explorez donc ces sessions.
6. **Mémoire élevée** : OpenReplay surveille la consommation de ressources de votre application afin de fournir des analyses à ce sujet. Mais en bonus, nous signalons également les sessions qui présentent une consommation de mémoire élevée par rapport à la moyenne de vos sessions enregistrées.
7. **CPU élevé** : Comme pour la mémoire, si l'utilisation du CPU d'une session particulière dépasse la moyenne calculée de toutes les autres sessions, nous la signalerons pour que vous puissiez l'examiner.
8. **Plantages** : Toute session dans laquelle l'application plante (devient inutilisable) sera signalée pour que vous puissiez l'examiner.

## Visualiser les problèmes pendant le replay d'une session

Ce qui est appréciable dans la mécanique du replay, c'est que les sessions ne sont pas seulement signalées avec ces erreurs : vous pouvez aussi les examiner pour comprendre exactement _quand_ elles se produisent et, en observant les actions de votre utilisateur, vous pouvez également découvrir le _pourquoi_ qui se cache derrière.

La capture d'écran suivante vous montre ces problèmes affichés dans différentes parties de l'interface du lecteur :



![](https://i.imgur.com/byIiNZO.png)

Comme vous pouvez le constater, juste sur la chronologie, vous avez tous les problèmes rencontrés pendant l'enregistrement de la session. Chaque problème est identifié par une icône différente.
Mais dans des cas comme celui de la capture d'écran, où ils sont trop nombreux et serrés les uns contre les autres, la liste des Événements Utilisateur à droite peut vous aider à identifier l'ordre dans lequel ils se produisent (cela dit, il ne s'agit que d'événements _utilisateur_, ce qui signifie que tout n'est pas enregistré ici).

En cliquant à la fois sur les icônes des chronologies ou sur les éléments de la liste des événements, vous sauterez au moment précis, au sein du replay, où cet événement a lieu.

Si, en plus de cela, vous utilisez des plugins pour enrichir le type de données capturées, comme [le plugin fetch](https://docs.openreplay.com/tutorials/fetch) pour capturer les informations des requêtes, vous pourrez inspecter ces problèmes encore plus en détail.
Jetez un œil à la capture d'écran suivante montrant les détails d'une requête ayant échoué :


![](https://i.imgur.com/97bREgW.png)

Remarquez le code de **statut 404** dans le coin supérieur droit de l'écran. Ainsi que le reste des informations de la requête qui se trouvent en dessous.
